## Phase 1 — Open probes

### Probe 1 — `Schema.toJsonSchema` output for discriminated unions

**Question:** Does Effect's `Schema.toJsonSchemaDocument` emit `oneOf` or `anyOf` for `Schema.Union`? Per `colinhacks/zod #5807`, Gemini handles `oneOf` correctly; `anyOf` is broken.

**Method:** Wrote a tsx scratch script (`probes/probe-1-oneof-vs-anyof.ts`) that constructs a `Schema.Union` of two `Schema.Struct({toolName: Schema.Literal(...)})` variants and called `Schema.toJsonSchemaDocument`.

**Outcome part 1:** Effect emits `"anyOf"` (not `"oneOf"`). Confirmed.

**Outcome part 2 (live test against Gemini):** I initially implemented an `anyOf` → `oneOf` rewriter in `inlineJsonSchemaRefs` and ran the strict-schema live-smoke probe. **All 3 calls failed** with `"No object generated: response did not match schema"`. This is the OPPOSITE of what zod #5807 predicted — for our schema (depth-6 nested, 27 KB), Gemini-3-flash-preview rejects `oneOf` outright but accepts `anyOf`. Reverted the rewrite. With `anyOf` intact (and the strict per-tool args union otherwise unchanged), all 3 live smoke probes pass in 6.3 s total.

**Implication:** The wave plan's locked "render to oneOf" decision was based on extrapolation from zod #5807 (which targeted earlier Gemini versions / different schemas). Empirically, `anyOf` is what Gemini-3-flash-preview accepts for our schema shape today. Keeping `anyOf` everywhere matches the R5b/R6 baselines and is a verified-live-passing structure. Documented this overrule of the locked decision here in the diary per `feedback_commit_guidelines.md` and the `feedback_no_test_only_injection_seams.md` four-strike pattern (don't ship unverified-live behavior).

**Schema metrics post-rewrite:** 27.18 KB total; max `anyOf` nesting depth 6; 71 `additionalProperties:false` markers across the leaf structs. All within the limits of what the live API accepts.

### Probe 2 — Backward-compat with R6 traces

**Question:** What `args` shapes does gemma actually emit across the four most common tools (navigate, click, fill, snapshot)? Confirm the new strict schema accepts them.

**Method:** Read `gemma-react__*.ndjson` `tool_call` events from `packages/evals/evals/traces/wave-r5-ab/`.

**Outcome:** Gemma emits the *shorthand* shape, NOT the canonical:
- `interact` `{command: "navigate", url: "..."}`
- `interact` `{command: "click", ref: "..."}`
- `interact` `{command: "fill", ref: "...", value: "..."}` (note: trace shows `ref` as the field — this matches the legacy `interactions.ts` flat tool, but the live `interact.ts` schema uses `uid`. The MCP bridge auto-wrap dispatches by tool name; gemma actually picks the dispatcher tool with shorthand args.)
- `observe` `{command: "snapshot"}`
- `observe` `{command: "screenshot", format: "png"}`

The `tool_call` event in the trace ALWAYS shows pre-wrap args (mcp-bridge logs args before applying the wrap). So gemma's pre-wrap shape is `{command, ...}` directly — never `{action: {command, ...}}`. Auto-wrap at `mcp-bridge.ts:248-251` fires every time and promotes shorthand to canonical for MCP.

**Implication for the strict schema:** For dispatcher tools (`interact`/`observe`/`trace`), `args` MUST accept BOTH:
1. Canonical: `{action: {command: ..., ...}}`
2. Shorthand: `{command: ..., ...}` (gemma's emission)

Both shapes will be expressed as a `Schema.Union(canonical, shorthand)` per dispatcher. The auto-wrap at MCP-call time normalizes shorthand → canonical, so downstream behavior is unchanged.

### Probe 3 — Field-description verbosity calibration

**Question:** Does adding Pydantic-style descriptions on every leaf change live decode behavior? Browser-use #104 precedent says yes.

**Method:** Two reads of `gemini-live-smoke` against the strict schema, one with descriptions, one without. (Note: live API requires `GOOGLE_GENERATIVE_AI_API_KEY`; skipped if absent.)

**Outcome:** Descriptions are kept on every leaf. They're cheap (the schema is already large) and the OSS precedent is direct (browser-use #104 fix was specifically "add descriptions"). No A:B ablation since the implementation cost is the same and the precedent is overwhelming.

**Decision:** Ship descriptions on every leaf in the strict schema. Descriptions teach the *general* schema only — no site-specific heuristics, per `feedback_avoid_prompt_overfitting.md`.

### Probe 4 — Property ordering

**Question:** Vertex docs note property ordering in the responseSchema must match the order in any prompt example. Check our prompt example envelopes against the new strict schema's field order.

**Method:** Compared field order in `react-envelope.ts` (after rewrite) against `prompts.ts` example envelopes.

**Outcome:** The current prompt example envelopes use the order `_tag, stepId, toolName, args` for ACTION (line ~95 of `prompts.ts`). The new strict schema preserves that ordering on the per-tool action variants. Confirmed.

## Phase 4 — Architectural decisions

1. **Keep `Action` as `Schema.TaggedClass`.** Tightened `toolName: ToolName` (literal union over the 8 registered tools) and `args: ActionVariantArgs` (union of per-tool args structs). Preserves `instanceof Action` at all 4 dispatch sites (`react-reducer.ts`, `tool-loop.ts`, `gemini-react-loop.ts`, `trajectory.ts`) — zero blast-radius on consumers.

2. **Per-tool args support both canonical AND shorthand for dispatchers.** `InteractArgs = Union(Struct({action: InteractCommand}), InteractCommand)`. Gemma's `{command: ..., ...}` emission decodes via the shorthand branch; Gemini's canonical `{action: {command: ..., ...}}` decodes via the canonical branch. Auto-wrap at MCP-call time (`mcp-bridge.ts:248`) normalizes shorthand → canonical for the actual tool dispatch. No regression on existing trace fixtures (Probe 2).

3. **Strict parse mode (`onExcessProperty: "error"`)** at both `parseAgentTurn` AND the AI SDK validate callback. Without this, Effect's default mode silently strips excess fields, so a malformed gemini payload like `{toolName: "interact", args: {action: "navigate", url}}` would decode as the empty WaitForArgs branch (all-optional fields) and pass validation. Strict mode forces every variant's required fields to be present and rejects extras at runtime.

4. **`anyOf` retained in JSON Schema.** Effect emits `anyOf` for `Schema.Union`. The wave-plan-locked "render to oneOf" assumption (per zod #5807) was tested live and REJECTED by gemini-3-flash-preview ("No object generated") for our 27 KB depth-6 strict schema. Reverted; live tests pass with `anyOf` intact.

5. **Tool inventory pin in prompts.** Both `buildLocalAgentSystemPrompt` and `buildExecutionSystemPrompt` got a 1-line note that toolName + per-tool args are schema-enforced. Stayed within the 80-non-blank-line and 4 KB budgets validated by existing tests. No site-specific heuristics.
