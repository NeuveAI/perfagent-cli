## Decisions made under ambiguity

1. **`_tag` discriminators.** PRD §3.3 sketch shows SCREAMING wire tags (`THOUGHT`,
   `ACTION`, `PLAN_UPDATE`, …). I used `Schema.TaggedClass<Foo>()("THOUGHT", { ... })`
   so the class identifier is itself the wire `_tag`. The TypeScript class name is
   PascalCase (`Thought`, `Action`, `PlanUpdate`, `StepDone`, `AssertionFailed`,
   `RunCompleted`); the `_tag` runtime value is SCREAMING.

2. **Two `PlanUpdate` symbols.** The wire variant in `react-envelope.ts` and the
   in-domain execution event in `models.ts` share the class name. They are
   intentionally distinct: wire-side `_tag === "PLAN_UPDATE"`, in-domain event
   `_tag === "PlanUpdate"` matching the existing CamelCase convention of sibling
   events (`StepStarted`, `ToolCall`, …). The test file imports them as
   `PlanUpdateTurn` and `PlanUpdateEvent` to disambiguate. R3's reducer will be
   the one site that bridges the two.

3. **Two `RunCompleted` symbols, too.** `models.ts` already had a `RunCompleted`
   class (a separate `WatchEvent` payload carrying a `PerfReport`). I added the
   ReAct envelope variant `RunCompleted` (with `status` + `summary`) in
   `react-envelope.ts`. They live in separate modules and the project follows
   "no barrel files," so consumers always import from the specific source. No
   collision in practice.

4. **`applyPlanUpdate` is synchronous, payload is `Schema.Unknown`.** Per the
   T2 description's "prefer (a) — keep the method synchronous." The class
   schema field is `payload: Schema.Unknown`, but the method's mutation logic
   uses `update.payload instanceof AnalysisStep` to gate insert/replace.
   **Round 2 update (per reviewer M1):** if the payload is not an `AnalysisStep`
   for `insert`/`replace`/`replace_step`, the method now `throw`s synchronously
   (CLAUDE.md "Unrecoverable Errors Must Defect"). This makes a buggy R3 reducer
   crash loudly instead of silently desyncing the trajectory and `steps`. The
   R3 reducer is responsible for decoding the wire payload into an
   `AnalysisStep` before constructing the `PlanUpdate` event. Three new tests
   pin the throw on insert / replace / replace_step with bad payloads.

5. **Insert positioning.** PRD says "append new `AnalysisStep` at position
   implied by stepId (or end if not found)." I interpreted "position implied by
   stepId" as "before the step whose `id === update.stepId`", so the new step
   takes that slot and the existing one shifts right. Falls through to "append
   at end" when no step matches. Test `insert: places the new step at the
index of the matching stepId…` pins this behavior.

6. **`replace` vs `replace_step`.** Routed through the same code path per task
   description ("alias from PRD sketch — keep both literals, route same code
   path"). Test `replace_step: behaves identically to replace` pins this.

7. **Immutability invariants.** `applyPlanUpdate` returns a new
   `ExecutedPerfPlan` with new `steps` and `events` arrays. Reference equality
   tests confirm `original.steps !== result.steps` and the original's snapshot
   is preserved.

## Round 2 patches (post-reviewer)

Reviewer round 1 returned `REQUEST_CHANGES` (2 major, 3 minor). All addressed:

- **M1 — `applyPlanUpdate` silent-drop on bad payload (`packages/shared/src/models.ts`):**
  Picked **Option A (defect)** per reviewer guidance. The branch that
  previously appended only the event and left `steps` unchanged now `throw`s
  with a descriptive message ("requires payload to be an AnalysisStep
  instance; got <type>; decode the wire payload upstream…"). Synchronous
  defect — equivalent to `Effect.die` for non-Effect code. Bad callers crash
  loudly instead of silently corrupting the plan.

- **M2 — Test coverage of the silent-drop / throw branch:** Added three
  tests (`tests/react-envelope.test.ts`):
  `throws when insert is called with a non-AnalysisStep payload`,
  `throws when replace is called with an undefined payload`,
  `throws when replace_step is called with a non-AnalysisStep payload`.
  Each uses `assert.throws(...)` with regex `/AnalysisStep instance/` to
  pin the message contract.

- **m3 — Diary inaccuracy:** "no-op" → "renders a real `tag: 'plan'` row
  with the action label and step ID" in the "What changed" table. The
  `raw-events-overlay.tsx` matcher branch is in fact a real renderer, not
  a placeholder.

- **m4 — ASSERTION_FAILED literal coverage:** Replaced the single
  `(budget-violation, perf)` round-trip case with a parametric loop over
  all 5 categories × 5 domains = 25 named tests
  (`decodes ASSERTION_FAILED with category=… domain=…`). Any future
  schema literal misspelling now fails its own focused test.

- **m5 — Pre-existing events preservation:** Added test
  `preserves all pre-existing events at their original indices through
  chained calls`. Starting from a plan with 1 `RunStarted` event, it
  applies `insert → replace → remove` and asserts the resulting events
  array length is `originalEvents.length + 3`, the original events are
  reference-equal at their original indices, and each new `PlanUpdate`
  appears at the expected post-original index.

- **Smaller items folded in:**
  - New test `preserves deeply-nested ACTION args verbatim through
    Schema.Unknown` exercises a 4-level nested object via
    `Schema.Unknown` round-trip.
  - New test `tolerates leading and trailing whitespace around the JSON
    payload` confirms `Schema.fromJsonString` strips whitespace before
    parsing.
  - `Effect.annotateCurrentSpan` on parser entry points: deferred to R2
    per reviewer ("non-blocking, R2's call").
