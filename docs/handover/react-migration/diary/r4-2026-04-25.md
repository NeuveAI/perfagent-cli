## T1 — Trajectory rolling data structure + rule-based summarization

### Goal

Pure functions that, given the full chat-history `messages` array accumulated by
`packages/local-agent/src/tool-loop.ts`, produce a "rolled" version where:

- The system message and the first user message (the run's task description) are
  preserved verbatim — they are the contract.
- The most recent **N=10** assistant/observation pairs are kept verbatim
  (PRD §4 Decision #6, JetBrains 10-turn sliding window).
- Older pairs are collapsed into a single rule-based `<trajectory_summary>`
  block whose body is one `<event>...</event>` line per turn
  (PRD §R4 line 260).

The summarizer is deterministic, side-effect free, and uses the canonical
`AgentTurn` schema to extract semantic fields (`_tag`, `toolName`, `summary`,
etc.) — no regex, no string-shape heuristics. Per
`feedback_types_over_regex.md` and CLAUDE.md "Prefer Schemas Over Fragile
Property Checks".

This is the foundation that T3 wires into the live tool-loop.

### Decisions made under ambiguity

1. **`TrajectoryMessage` is a plain `interface`, not a `Schema.Class`.** The
   message shape (`role` + `content`) is owned by `@neuve/local-agent`'s
   `OllamaMessage` interface — also a plain interface. Re-modelling it as a
   schema in `@neuve/shared` would force a circular layering. Keep the
   trajectory module structurally compatible: any object that satisfies
   `{ role, content }` (including `OllamaMessage`) is accepted. T3 imports
   `OllamaMessage[]` and passes it directly without a wrapper conversion.

2. **`partitionTrajectory` returns `{ preface, turns, trailing }` instead of
   collapsing trailing dangling messages.** The chat history is always
   well-formed at the moment of `rollTrajectory` invocation (T3 calls it just
   before each Ollama chat — never mid-pair). But the partition primitive is
   exported separately for tests and possible future consumers, and a
   mid-pair input shouldn't crash. The rolling pipeline appends `trailing`
   verbatim at the very end so the input is always recoverable.

3. **Summarizer head format is one of six fixed shapes per `_tag`** — chosen
   so each line is self-describing without re-reading the original envelope:
   - `THOUGHT step-id` (stepId carries semantics)
   - `ACTION toolName` (toolName is the call signal; stepId would be redundant
     when the prior STEP_DONE / next STEP_DONE bracket the call)
   - `PLAN_UPDATE action stepId` (action is the dominant signal)
   - `STEP_DONE stepId`
   - `ASSERTION_FAILED stepId (category)` (category in parens — consumers
     scanning for budget violations want it visible)
   - `RUN_COMPLETED status` (final marker)

4. **Outcome separator is " → " (en-arrow-space).** Matches PRD §R4 line 260
   exactly: `<event>TOOL action → outcome</event>`. ASCII alternative
   considered (`->`) and rejected — the arrow is in the PRD verbatim.

5. **Empty observation produces `<event>HEAD</event>` (no arrow), not
   `<event>HEAD → </event>`.** Cleaner, fewer dangling spaces, easier for
   the model to scan. Tested explicitly.

6. **Whitespace collapse + truncate happens AFTER channel stripping.** The
   stripped block can leave double spaces (e.g. `head  middle  tail`) — the
   collapser reduces runs of whitespace to a single space, normalizes
   newlines/tabs to space, and trims trailing space. Done in a single linear
   scan (no regex, no `.replace`). Truncation at the configured char limit
   appends `…` (one Unicode ellipsis character, not three dots).

7. **Decode failure becomes `<event>UNPARSED → outcome</event>`.** A
   format-grammar-enforced AgentTurn JSON should never fail to parse in
   production, but we cannot let a corrupt entry crash the whole rolling
   pipeline. The fallback line preserves the observation outcome so the
   model still has the post-step context.

## T2 — Strip `<|channel>thought…<channel|>` tokens from historical text

### Goal

Defensive, forward-looking pure helper used by T1's trajectory summarizer
(and reused by T3's prompt-assembly pipeline) so any historical assistant
text containing Gemma 4's channel-tagged thinking is reduced to the user-
visible response only. Per Gemma 4 model card and PRD §R4 line 269.

### Files changed

| File | Kind | Purpose |
|------|------|---------|
| `packages/shared/src/strip-thought-channel.ts` | new | Single exported `stripThoughtChannel(text: string): string`. Pure. Uses `String.prototype.indexOf` + `slice` only. |
| `packages/shared/tests/strip-thought-channel.test.ts` | new | 10 unit tests covering: no-op (no markers); empty input; single block (PRD example); middle-of-message block; consecutive blocks; non-consecutive blocks; dangling open without close (slice off); dangling close without open (preserved as literal); empty channel block; bare open at end. |

### Decisions made under ambiguity

1. **Delimiter pair: `<|channel>` open, `<channel|>` close — the literal
   asymmetric form spelled out in PRD §R4 line 269.** This is unusual (most
   templating uses symmetric `<|x|>` pairs) but matches the canonical Gemma 4
   tokenization the PRD references. Tests pin both forms.

2. **Dangling open without close → drop everything from open to end of
   input.** Conservative: a leaked open token implies the response was
   truncated mid-think; the rest is unrecoverable model-internal scratchpad.
   Tested explicitly.

3. **Dangling close without matching open → preserved verbatim.** A close
   token alone is just text — no scratchpad to strip. Tested explicitly.

4. **No regex.** `String.prototype.indexOf` is the only string-search
   primitive used. Iteratively scans the input once; `O(n × k)` worst case
   where `k` is the number of channel blocks. Per
   `feedback_types_over_regex.md`.

### Compliance audit (CLAUDE.md)

- ✅ Pure function — no Effect wrapping, no I/O.
- ✅ No regex. No `try`/`catch`. No `null`.
- ✅ Single focused utility per file (`utils/`-style; placed in `src/` since
  it is part of the shared public API consumed by `trajectory.ts`).
- ✅ kebab-case filename.

## T3 — Wire trajectory rolling into the live tool-loop

### Goal

Make every Ollama chat call see a bounded prompt: system + initial user +
optional summary block + last 10 verbatim assistant/observation pairs. The
caller-owned `messages` array continues to grow with the full trajectory; we
only slim the *view* sent to Ollama.

### Files changed

| File | Kind | Purpose |
|------|------|---------|
| `packages/shared/package.json` | edit | Add `./trajectory`, `./strip-thought-channel`, `./constants` to the `exports` map so consumers can import without barrels. |
| `packages/local-agent/src/tool-loop.ts` | edit | Import `rollTrajectory` from `@neuve/shared/trajectory`; before each `ollamaClient.chat(...)` call, compute the rolled view and pass `rolled.messages` instead of `messages`. The full `messages` array continues to accumulate. Log line extended with `sentMessageCount`, `summarizedTurns`, `verbatimTurns`. |
| `packages/local-agent/tests/tool-loop-trajectory.test.ts` | new | 2 integration tests on `runToolLoop`: (a) drives 14 turns, asserts the late-round chat call carries a `<trajectory_summary>` block and a 20-message verbatim tail, plus that the caller-owned `messages` keeps growing past the rolled view; (b) drives 6 turns (5 ACTION + RUN_COMPLETED), asserts no summary block in any chat call (turns below window). |

### Decisions made under ambiguity

1. **Rolling lives in `tool-loop.ts`, not `prompts.ts`.** PRD §R4 line 259-261
   nominally puts `<trajectory>` in `buildExecutionPrompt`. But our agent
   boundary is: supervisor calls the agent ONCE with an initial prompt; the
   agent owns the multi-turn loop and the chat history. `buildExecutionPrompt`
   produces the initial user message — it has no access to the live
   trajectory because that lives in the agent's process. Re-prompting from
   the supervisor every turn would be a different (and far more invasive)
   architecture. So the rolled trajectory belongs at the boundary where the
   chat history is mutated: `runToolLoop`. The PRD's `<trajectory>` tag is
   mapped onto our equivalent `<trajectory_summary>` user-role message
   (synthesized by `rollTrajectory`).

2. **Roll on every iteration, not just when the window is exceeded.**
   `rollTrajectory` is cheap (single pass over the messages array) and
   is a no-op when the turn count is at-or-below the window. Calling it
   unconditionally avoids any stateful "we've-already-rolled" bookkeeping.

3. **Caller-owned `messages` keeps the full trajectory.** Two reasons: (a)
   the post-loop session caller (`agent.ts`) saves `session.messages =
   messages` for replay/inspection — chopping it would lose history; (b) the
   roll is computed fresh each iteration, so persisting the full history
   means we can change the window size or stripping logic without losing
   prior context.

4. **Image budget / multimodal config deferred.** PRD §R4 also mentions a
   per-call SOM image budget (default 280 tokens). The current
   `OllamaMessage` interface is text-only (`role` + `content`); no image
   bytes ever enter the chat history. When multimodal input lands, image
   handling slots in adjacent to the text trajectory. Not blocking R4.

5. **Three new exports added to `@neuve/shared/package.json`** —
   `./trajectory`, `./strip-thought-channel`, and `./constants`. The constants
   export is needed because R4-T4 will reuse the supervisor-side constants
   pattern for budget thresholds. Per CLAUDE.md "No Barrel Files" each is a
   direct file path, no re-export hub.

## T4 — Budget monitor: warn at 96K, abort at 120K

### Goal

Inspect every `usage_update` event and:
- Log a single `react-budget-exceeded(level=warn)` warning the first time
  prompt tokens cross 96K.
- Synthesize an abort `RunFinished({ status: "failed", abort: { reason:
  "context-budget-exceeded" } })` when prompt tokens cross 120K — the
  existing R3 adherence gate accepts abort-marked RunFinished events
  (`executor.ts:183`), so the run terminates cleanly rather than being
  filtered as premature.

The warn/abort thresholds are pinned to PRD §R4 (75% / 93.75% of Gemma 4 E4B's
128K context).

### Files changed

| File | Kind | Purpose |
|------|------|---------|
| `packages/supervisor/src/constants.ts` | edit | Add `REACT_BUDGET_WARN_TOKENS = 96_000` and `REACT_BUDGET_ABORT_TOKENS = 120_000`. |
| `packages/supervisor/src/react-reducer.ts` | edit | Add `budgetExceeded: Schema.Boolean` to `ReactRunState` (initial=false). Extend `ReducerSignal` Data.TaggedEnum with a 5th variant `BudgetExceeded({ level, promptTokens, threshold })`. Add `ReducerSignalLevel = "warn" \| "abort"` literal type. |
| `packages/supervisor/src/budget-monitor.ts` | new | Pure `evaluateBudget(promptTokens, runState) → { signals, runState, shouldAbort }`. Branches: ≥abort threshold → abort signal + abort flag; ≥warn threshold AND not yet warned → warn signal + flag. Otherwise no-op (returns same `runState` reference). |

| `packages/supervisor/tests/executor-react-mode.test.ts` | edit | 3 new R4 integration tests: (a) warn-once across 3 usage_updates straddling threshold; (b) abort triggers context-budget-exceeded RunFinished and stream halts before subsequent agent_turn; (c) abort flows through adherence gate even with an intervening unresolved StepFailed. |

### Decisions made under ambiguity

1. **`ReducerSignal.BudgetExceeded` lives in `react-reducer.ts` even though
   `evaluateBudget` lives in `budget-monitor.ts`.** The signal union is the
   shared logging contract — keeping every variant in one place lets
   `logReducerSignal` exhaustively dispatch via if/else chain (the existing
   R3 pattern). `budget-monitor.ts` imports the type and ctor; the reducer
   itself never produces this variant. This is a clean split: state machine
   on AgentTurn lives in `react-reducer.ts`, state machine on usage_update
   lives in `budget-monitor.ts`, both share the signal vocabulary.

2. **Warn-once guard is a stateful flag, not a high-watermark check.** The
   alternative is "emit warn whenever promptTokens > previous max ≥ warn
   threshold" — but that requires tracking a watermark per run. Simpler:
   set `budgetExceeded=true` on the first crossing; suppress further warn
   emissions while it stays true. Above the abort threshold, we always emit
   regardless because that's a structural failure event (the run is
   terminating). Consequence: a run that briefly dips below warn and crosses
   again will NOT re-emit. Acceptable: a single warn per run is the
   operational signal; a noisier signal would clutter logs.

3. **Abort triggers a synthesized `RunFinished` event in the executor
   accumulator, NOT an exception or stream interrupt.** Reasons: (a) the
   existing `Stream.takeUntil((executed) => executed.hasRunFinished)`
   guarantees stream termination once a RunFinished exists — no separate
   abort plumbing needed; (b) the existing adherence gate (`executor.ts:183`)
   already accepts abort-marked RunFinished events; (c) downstream
   consumers (UI, eval harness) treat this RunFinished identically to an
   agent-emitted `RUN_COMPLETED|failed` with `category=abort`. Only the
   `summary` text and `abort.reason` distinguish a budget abort from any
   other.

4. **Prompt tokens come from `update.promptTokens` getter (already
   defined on `AcpUsageUpdate`).** The getter unwraps
   `_meta.promptTokens ?? undefined`; we coerce missing values to 0 so the
   evaluation runs anyway (zero is below both thresholds). Adapters that
   don't emit token telemetry (Gemini CLI, Claude Code) effectively
   short-circuit through the no-op branch.

5. **The budget branch is mutually exclusive with `addEvent(part)` for
   `usage_update`.** Previously `addEvent` was called on every
   sessionUpdate including `usage_update` — for usage_update it was a no-op
   (no event appended). Now we shortcut directly: the budget branch
   replaces the addEvent call for usage_update updates. No semantic change
   in the no-abort path; abort path adds the synthesized RunFinished.

## T5 — Tests + calibration probe + cross-package verification

### Goal

Roll up all R4 tests, add a synthetic-scale calibration probe that
demonstrates the rolling pipeline's bound on prompt growth, and verify the
8-package consumer set still typechecks + tests cleanly.

### Files changed

| File | Kind | Purpose |
|------|------|---------|
| `packages/shared/tests/trajectory-calibration.test.ts` | new | Synthetic-scale calibration probe with realistic envelope/observation sizes drawn from Q9 rebaseline data. 4 tests pin: (a) 60-turn trajectory rolls to <60% of unrolled size and stays under warn; (b) 250-turn trajectory still under warn (proves bound is independent of turn count); (c) summary block char-line bound matches the configured limit; (d) monotonicity — longer trajectories never produce smaller rolled prompts than shorter ones. |

### Decisions made under ambiguity

1. **Calibration is a synthetic probe, not a live Gemma replay.** A real
   replay against `gemma4:e4b` would (a) be slow, (b) hit the same MCP /
   browser stack as R3's Volvo replay (already validated), (c) be
   non-deterministic for CI assertions. The probe instead reproduces
   realistic content sizes: ~180-char ACTION envelopes, ~80-char
   `interact` observations, ~8KB `trace analyze` observations every 5th
   turn. These sizes come from the Q9 rebaseline trajectory dump (which
   showed peakPromptTokens 8K-30K at 11-15 turns with the executor
   accumulating insight outputs).

2. **Char→token estimate uses 4 chars/token.** Standard English
   approximation; Gemma 4's tokenizer averages slightly tighter
   (~3.5 chars/token for technical content), so our estimate is
   conservative — the probe's bound is a worst-case upper bound. Per
   `feedback_types_over_regex.md` and the Q9 Probe D finding (`promptEvalCount=15639` for ~62KB of context), the estimate is realistic.

3. **The probe's first claim is "rolling produces ≥40% reduction" rather
   than "unrolled exceeds abort threshold" because the synthetic content
   sizes don't naturally exceed 120K at 60 turns.** A more aggressive
   fixture (every-turn 8KB observations) would, but that's outside the
   observed Q9 distribution. The 60% upper-bound assertion catches
   regressions where rolling silently degrades, without overclaiming
   what the synthetic setup proves. The 250-turn case is included
   precisely to demonstrate the asymptotic bound — at 250 turns the
   unrolled view DOES exceed the warn threshold, and the rolled view
   stays well under.
