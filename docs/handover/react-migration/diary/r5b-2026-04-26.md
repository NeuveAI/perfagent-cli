## What broke

The R5 wave registered three eval suites in `packages/evals/evals/wave-r5-ab.eval.ts`:
`gemma-react` (production), `gemini-react` (frontier baseline), and
`gemma-oracle-plan` (ablation). The first sweep showed gemma-react at 25%
completion (the Q9 baseline floor) and **both Gemini-backed lanes at 0% — 40 of
60 evals failed in <2s each**. Two distinct bugs hid behind R5's
`MockLanguageModelV4`-only test coverage.

### Bug 1 — `wave-r5-ab.eval.ts` never loaded `.env.local`

`online-mind2web.eval.ts` loads dotenv at module top via `dotenv.config({ path: .env.local })`,
but `wave-r5-ab.eval.ts` (added by R5-T5) skipped that block. Result: every
`gemini-react` and `gemma-oracle-plan` task fast-failed with
`GeminiReactConfigError` / `PlannerConfigError` because
`GOOGLE_GENERATIVE_AI_API_KEY` was unset in the eval process's `process.env`
(it's gitignored in `packages/evals/.env.local`).

The lead applied the dotenv fix on disk before assigning T1 — already present
when I claimed the task. I verified it matches `online-mind2web.eval.ts:1-3,21-27` exactly.

### Bug 2 — Google rejects `$ref`/`$defs` in `responseSchema`

After bug 1 was fixed, the 2nd sweep showed gemini-react still 0% — but with
no error logs. The eval log went silent for the full 600s vitest test timeout
on every gemini-react task. The smoking gun in `run-2.log` is that the very
last visible log line is `Gemini-react session started …` and then nothing
until the harness force-killed the run.

Root cause: `gemini-react-loop.ts:62-77` (pre-fix) constructed
`AGENT_TURN_JSON_SCHEMA` as `{ ...document.schema, $defs: document.definitions }`
where `document` is `Schema.toJsonSchemaDocument(AgentTurn)`. That output is
`{anyOf: [{$ref: "#/$defs/THOUGHT"}, {$ref: "#/$defs/ACTION"}, ...], $defs: {THOUGHT: {...}, ACTION: {...}, ...}}`.

Google's Gemini `responseSchema` is an OpenAPI 3.0 subset and **rejects `$ref`
references**. The `generateObject` call throws "No object generated" in <2s on
every call. The throw is caught into a `GeminiReactCallError`, then bubbled
into an `AcpStreamError` in `gemini-agent.ts` — and at no point along that
path was the failure logged at INFO/ERROR. The supervisor's stream
machinery then waited politely for further updates that never came, hitting
the 600s timeout. The probe in `/tmp/wave-r5-ab/probe-flatten.ts` confirmed
the fix path: when each `$ref` is replaced inline by the resolved
`definitions[name]` object, both `gemini-3-flash-preview` and
`gemini-2.5-flash` return valid envelopes in <2s.

### Why it took so long to find

`MockLanguageModelV4` from `ai/test` (used by `gemini-react-loop.test.ts`)
ignores the schema — it returns whatever bytes the test feeds it via
`doGenerate`. Live Gemini does not. R5 shipped 4 unit tests covering happy
path / PLAN_UPDATE / doom-loop / schema-violation, all passing on mocks; the
production schema was never round-tripped against the real provider. The R5b
wave's mandate is to add a non-mock regression guard so this strike is the
last one (per `feedback_no_test_only_injection_seams.md` step 5).

### Architectural decisions

1. **Inliner location: keep inline in `gemini-react-loop.ts`, do not extract a
   shared util.** Cross-check via `grep -rn "Schema.toJsonSchemaDocument" packages/`
   surfaced exactly two call sites:
   - `packages/evals/src/runners/gemini-react-loop.ts` (this file — Google).
   - `packages/local-agent/src/tool-loop.ts:36-39` — also builds
     `{...document.schema, $defs: document.definitions}` but ships it to **Ollama**
     as the `format` grammar. Ollama's grammar engine tolerates `$defs`/`$ref`
     (it's documented to handle JSON Schema 2020-12), and the live gemma-react
     traces from R5 confirm tool-loop runs without issue.

   Per the task spec ("Keep it minimal — only extract if there are 2+ call
   sites"), there is **one Google-bound call site**. Extracting a shared
   `flatten-json-schema-refs.ts` module for one consumer would be premature.
   If a future ReAct backend (e.g. OpenAI structured outputs, which has
   different schema rules again) lands, that's the right time to extract.

2. **Test imports the live schema constant directly, not the runner.** The
   regression we're guarding is the schema → Google round-trip; the loop
   wiring is already covered by the four mock-backed unit tests. A test that
   only exercises `runGeminiReactLoop` end-to-end would also need an MCP
   bridge fake, would test more wiring than schema validity, and would be
   slower. Importing `AGENT_TURN_RESPONSE_SCHEMA` and calling
   `generateObject` directly with it gives the tightest possible regression
   coverage: any future change that breaks Google's schema acceptance fails
   this test in <30s.

3. **Logging via `Effect.tapError`, not `Effect.catchTag`.** The task spec
   asked to make the failure visible without changing the failure mode.
   `tapError` runs the side-effect (logging) on the failure channel and then
   re-fails — exactly what's needed. `catchTag` would recover, which would
   change behavior. The catch callback inside `Effect.tryPromise` is a sync
   error transformer that can only return errors; it cannot run an Effect
   like `Effect.logError`, so `tapError` on the surrounding pipe is the
   right surface.

4. **`Predicate.isObject` over manual type guards or `as` casts.** Effect's
   `Predicate.isObject` narrows `unknown` to `{ [x: PropertyKey]: unknown }`,
   letting me index `schema["$ref"]` without a cast. The walker stays pure
   and easy to read, satisfying CLAUDE.md's "no `as` casts unless
   unavoidable" rule.

## Outstanding / for reviewer

Reviewer (T2) should challenge:
- Whether the inliner should be exported as a shared util (decision: no —
  one Google call site; document in §"Architectural decisions" §1).
- Whether `Effect.tapError` is the right pattern vs. moving the
  `tryPromise.catch` into a separate Effect that explicitly logs and then
  re-fails (decision: no — `tapError` is the canonical surface for "observe
  the error, don't recover").
- Whether the smoke test should also exercise `runGeminiReactLoop` end-to-end
  with an MCP-bridge fake (decision: no — that's loop wiring coverage which
  the four existing mock tests already give; the schema → Google round-trip
  is the regression-prone surface).
- Whether the single `as JSONSchema7` cast should be eliminated (decision:
  no — it bridges two third-party schema specs, can't be avoided without
  forking one of the libraries).
