### Decisions made under ambiguity

1. **Public Ollama-native types over OpenAI-shape compat layer.** I considered keeping the
   `ChatCompletionMessageParam[]` / `ChatCompletionTool[]` boundary so tool-loop.ts could be
   touched minimally. Rejected because: (a) the tool-loop is being rewritten in T2 anyway,
   (b) preserving OpenAI types would have meant a translation layer that exists for one
   wave then gets ripped out ‚Äî wasted work, and (c) keeping the OpenAI types created
   subtle gaps (e.g. `tool_call_id` doesn't exist in native /api/chat tool-result messages
   so the field would have been ignored anyway). The minimal-disruption translation is in
   tool-loop's `annotateToolCalls` which synthesizes a synthetic `tool_call_id` per call
   so the existing ACP session-update wiring keeps working.

2. **`createOllamaClient` stays a synchronous factory.** `LocalAgent` is a non-Effect
   class wired through ACP. Forcing a `ManagedRuntime` here would clash with CLAUDE.md's
   "Do not use `ManagedRuntime` when you are already inside Effect code" guidance, and
   require a layer setup the package doesn't have. Instead, the factory uses
   `Effect.runSync(resolveStartupConfig)` to read `Config.string` env vars at construction
   time. Since both configs use `Config.withDefault`, the resolution can never fail. The
   per-call `chat` method returns Effects so callers integrate cleanly with Effect chains
   (T2's tool-loop will be Effect-native).

3. **NDJSON line buffering with explicit trailing-flush.** Ollama's `/api/chat` with
   `stream: true` emits one JSON document per line, with a final `done: true` line carrying
   token counts. The reader buffers chunks, splits on `\n`, decodes each complete line via
   `Schema.decodeEffect(Schema.fromJsonString(OllamaChatChunk))`. On stream end, any
   remaining non-empty buffer is flushed and decoded too ‚Äî defensive against servers that
   omit the trailing newline.

4. **Schema-decode failures `Effect.die`, not domain errors.** Per CLAUDE.md "Infrastructure
   errors become defects" and "Unrecoverable Errors Must Defect": a schema-decode failure
   on an Ollama NDJSON chunk means either Ollama's wire protocol diverged from our schema,
   or our schema has a bug. Either way the right move is loud crash. Not wrapping into
   `OllamaStreamError` (which would silently surface as a recoverable domain error to T2).

5. **Token-usage shape preserved end-to-end.** Native /api/chat exposes
   `prompt_eval_count` and `eval_count` (Ollama-side counts). The existing `usage_update`
   ACP session update expects `promptTokens`/`completionTokens`/`totalTokens`; we map
   `prompt_eval_count ‚Üí promptTokens`, `eval_count ‚Üí completionTokens`, sum to `totalTokens`.
   No information lost.

6. **`Config.string` reads only at construction time.** I considered re-reading per
   `chat` call so test harnesses could swap URLs. Rejected ‚Äî it would invite the
   "no test-only injection seam" anti-pattern (per `feedback_no_test_only_injection_seams.md`).
   Real callers set the env var before launching the local-agent process; tests live in
   `tests/dist-spawn.test.ts` which spawns the binary as a subprocess and can pass
   env explicitly.

### Decisions made under ambiguity

1. **`Schema.toJsonSchemaDocument` shape vs Ollama's `format` shape.** `toJsonSchemaDocument`
   returns `{ dialect, schema, definitions }` with `$ref`-based unions. Ollama's `format`
   parameter expects a single JSON Schema object. I combine them into
   `{ ...document.schema, $defs: document.definitions }` ‚Äî canonical draft-2020-12 idiomatic
   inline schema with $defs. llama.cpp's grammar generator follows `$ref`/`$defs` so
   the discriminated union is preserved verbatim. Cached in module-scope `AGENT_TURN_FORMAT`
   constant; not regenerated per call. Confirmed by a manual `node` probe that the
   produced shape has `anyOf` of `$ref` to the six `$defs` (THOUGHT, ACTION, PLAN_UPDATE,
   STEP_DONE, ASSERTION_FAILED, RUN_COMPLETED).

2. **Observation feedback as `role: "user"` with `<observation>...</observation>`.**
   Native /api/chat supports a `role: "tool"` message but provides no `tool_call_id`
   linkage. Since Variant B doesn't use native `tool_calls`, the MCP-side response is
   logically a tool result fed back to the agent. I chose `role: "user"` with an XML
   `<observation>` wrapper because the system prompt (R2-T3 next) will teach Gemma to
   read that block. This keeps message-shape uniform across THOUGHT (no real
   observation, just "proceed") / PLAN_UPDATE / STEP_DONE / ASSERTION_FAILED / ACTION,
   so the model sees a consistent pattern. Alternative considered: emit `role: "tool"`
   only after ACTION; rejected as it would split the per-envelope feedback path.

3. **Dispatch via `instanceof` on TaggedClass instances.** `parseAgentTurnFromString`
   returns class instances (Effect's `Schema.TaggedClass` decoder builds the class).
   `instanceof` narrows correctly; alternative was discriminated-union narrowing on
   `_tag`. Chose `instanceof` because the resulting code reads as "this envelope IS a
   THOUGHT" rather than "this envelope's tag is THOUGHT", and it matches what R3's
   reducer will likely look like.

4. **Schema-decode failure path: terminate the loop with an explanatory ACP message.**
   The task description suggests "emit a `RunFinished` with `category=abort` reason
   `non-schema-valid agent output`". `RunFinished` is an executor-level event, not a
   tool-loop event ‚Äî that's R3's domain. The closest tool-loop equivalent is an
   `agent_message_chunk` with the category-abort reason text. The text mirrors what
   the R3 adherence gate will surface, so the executor-side handling can pattern-match
   on the message later if needed (or R3 can introduce a dedicated session-update kind
   and replace this).

5. **Doom-loop detector keys off ACTION envelopes only.** Pre-T2 the detector checked
   "3 identical consecutive tool calls". Post-T2 it's "3 identical consecutive ACTION
   envelopes" ‚Äî `recentCalls` is mutated **only inside the ACTION branch**: a
   non-matching ACTION (different toolName or argsHash) clears the buffer, a matching
   ACTION pushes onto it, non-ACTION envelopes (THOUGHT / PLAN_UPDATE / STEP_DONE /
   ASSERTION_FAILED) leave the buffer untouched. So a sequence like
   `ACTION_X ‚Üí THOUGHT ‚Üí ACTION_X ‚Üí THOUGHT ‚Üí ACTION_X` still trips the detector at
   the third ACTION_X ‚Äî non-action envelopes don't reset the count. This matches the
   intent: identical ACTION envelopes *with intervening reasoning* are still a doom
   loop. The threshold (`DOOM_LOOP_THRESHOLD=3`) stays unchanged.

6. **Empty `result.content` fast-path.** With format grammar applied, the model
   physically cannot emit empty content; if it does, that's a server-side cancel or
   timeout. We emit a clear ACP message and exit rather than silently looping at the
   parse step. Matches the pre-T2 "model returned empty response" handler.

7. **Did NOT delete `repairAndParseJson`.** The task asks me to confirm it's gone ‚Äî
   `git grep` for that symbol returns nothing across the workspace (it was already
   removed in `b0ae25b1`, per the task description). Confirmed.

### Compliance audit (CLAUDE.md)

- ‚úÖ Parser via `Schema.decodeEffect` only ‚Äî `parseAgentTurnFromString` from R1. Zero regex anywhere in dispatch path.
- ‚úÖ No `Effect.catchAll` ‚Äî only `Effect.catchTag("SchemaError", ...)` to recover into a typed sentinel for parse-error abort.

### Decisions made under ambiguity

1. **`EVAL_GEMMA_PLANNER=template` instead of `oracle-plan`.** The lead's task description
   said "use `EVAL_GEMMA_PLANNER=oracle-plan`". `oracle-plan` calls Gemini Flash 3 via

   no `.env` file at the repo root). First attempt with `oracle-plan` failed all 5 tasks
   at the planner stage with `Oracle planner not configured: GOOGLE_GENERATIVE_AI_API_KEY
   is unset`. I switched to `template` mode, which uses the local rule-based decomposer
   (no external API). This keeps the rest of the pipeline integration test honest: the
   format-grammar + AgentTurn parser + tool-loop + native /api/chat all run end-to-end

### M1 ‚Äî Loud-but-harmless `/v1` strip (`packages/local-agent/src/ollama-client.ts`)

  who set the env to `http://localhost:11434/v1/` will see the warning in the log
  stream and can decide whether to migrate. Option A preserves backward compat for any

  the `/v1` detection + `Effect.logWarning` happens inside `resolveStartupConfig`'s
