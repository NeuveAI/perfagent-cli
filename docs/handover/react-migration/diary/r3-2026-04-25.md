## T1 — `packages/supervisor/src/react-reducer.ts` (foundation)

### Goal

Build a pure-ish state machine that consumes `AgentTurn` envelopes (wire shape
from R1 `packages/shared/src/react-envelope.ts`) and produces the updated
`ExecutedPerfPlan` plus any control signals (REFLECT trigger, PLAN_UPDATE cap
exceeded, premature RUN_COMPLETED, invalid PLAN_UPDATE payload). The reducer
performs no I/O — no MCP calls, no LLM calls, no file/DB access. It is the
engine that T2's executor and T4's tests share.

### Decisions made under ambiguity

1. **`ReactRunState` is a `Schema.Class`, fields are: `planUpdateCount: Number`,
   `consecutiveAssertionFailures: Record<string, number>`, `lastTurnTag: optional
   String`.** Per task description and CLAUDE.md "Schema.Class for value objects".
   The `consecutiveAssertionFailures` record uses `Schema.String` keys (not
   branded `StepId` keys) for ergonomic spread/access — at the reducer interface
   the keys ARE real StepIds; we don't validate the record structure since the
   reducer is the only writer. A `static readonly initial` exposes the empty
   starting state for executors and tests. There is no `update(fields)` helper
   — each branch constructs a fresh `new ReactRunState({...})` to keep the
   immutability contract obvious.

2. **`ReducerSignal` is a `Data.TaggedEnum` with FOUR variants (not three).**
   The task description's three variants (`ReflectTriggered`,
   `PlanUpdateCapExceeded`, `PrematureRunCompleted`) are illustrative ("variants
   like..."). I added a fourth — `InvalidPlanUpdatePayload` — for the case where
   Gemma emits a PLAN_UPDATE envelope whose `payload: Schema.Unknown` does not
   decode to a valid `AnalysisStep`. The wire schema does not constrain the
   payload (it is `Schema.Unknown` per R1), so the format-grammar contract does
   not catch a malformed step body — that is a domain-level recoverable error,
   not a defect. The reducer must emit a signal AND skip mutation. Defecting via
   `Effect.die` would crash the run on a Gemma quality issue (vs. a true bug);
   that is the wrong shape per CLAUDE.md "Reserve the error channel for
   recoverable, expected errors". The `InvalidPlanUpdatePayload` signal carries
   `stepId`, `action`, and the decode `cause` so the executor can decide
   retry/replan/abort.

3. **Reuse existing `StepFailed` for ASSERTION_FAILED — do not extend models.ts.**
   The task description asks "What event class to use for ASSERTION_FAILED — new
   class or reuse `StepFailed`?". I chose reuse without extending. `StepFailed`
   already carries `stepId`, `message`, optional `category`, optional `abortReason`
   — all the fields the existing harness/UI/adherence gate consume. The wire
   envelope's additional `domain` and `evidence` fields get packed into the
   `message` string as a structured prefix
   `category=<c>; domain=<d>; reason=<r>; evidence=<e>` — round-trippable via the
   existing `parseAssertionTokens` logic in `models.ts` (reads `category=` and
   `abort_reason=`, ignores unknown keys). Keeping this lossless on the wire side
   would mean extending `StepFailed` schema (touching every consumer) for no
   immediate adherence-gate or UI need; reviewer can flag if this becomes load
   bearing later. Documented for T3's gate to verify the message-prefix shape
   round-trips.

4. **THOUGHT envelope appended as `AgentThinking` event with `text = envelope.thought`.**
   The existing `AgentThinking` class has only a `text: string` field — no
   stepId. I considered prepending `[<stepId>] ` to the text but that is a
   display concern, not a reducer concern. The reducer keeps the THOUGHT text
   verbatim; downstream consumers can correlate via event ordering with adjacent
   StepStarted/StepCompleted markers.

5. **ACTION envelope appended as `ToolCall` event.** The wire envelope's `args:
   Schema.Unknown` matches `ToolCall.input: Schema.Unknown` directly — no decode
   step needed. The reducer always appends the ToolCall (no de-duplication
   against a "last event === same ToolCall" check). T2's executor MUST ensure
   that the gemma-react path does NOT also call `plan.addEvent(tool_call ACP
   update)` for the same envelope — otherwise we would double-count. The reducer
   becomes the single source of plan mutation for gemma-react mode in T2.

6. **PLAN_UPDATE cap is hard, evaluated BEFORE mutation.** When
   `runState.planUpdateCount >= REACT_PLAN_UPDATE_CAP (5)`, the reducer emits
   `PlanUpdateCapExceeded` signal AND returns the unchanged plan (no event
   appended, no step mutation). Counter still increments by 1 so the executor
   can see "the model attempted 7 PLAN_UPDATEs total" via the runState. A
   subsequent PLAN_UPDATE will continue to emit cap-exceeded signals each time.

7. **PLAN_UPDATE payload decoding via `Schema.decodeUnknownEffect(AnalysisStep)`.**
   For `action ∈ {insert, replace, replace_step}` we decode the payload to a
   real `AnalysisStep` instance and pass it to R1's `plan.applyPlanUpdate(...)`
   (which throws synchronously on a non-AnalysisStep payload — defect). For
   `action === "remove"`, no decode needed — `applyPlanUpdate` handles it directly.
   On decode failure (`SchemaError`), we catch via `Effect.catchTag("SchemaError",
   ...)`, emit `InvalidPlanUpdatePayload`, and return the unchanged plan +
   incremented counter (decode-failed PLAN_UPDATEs still count toward the cap so
   a model spamming malformed payloads can't escape the cap).

8. **REFLECT triggers when `consecutiveAssertionFailures[stepId] >= 2` after
   incrementing on the current ASSERTION_FAILED.** Fires every time the count is
   ≥2 (i.e., on the 2nd, 3rd, 4th... consecutive failure). The executor / harness
   can de-dupe the signal in T2 if needed. STEP_DONE on the same stepId resets
   the counter to 0 (preserving the key in the record but clearing the count) —
   future failures restart the chain. ASSERTION_FAILED on a different stepId does
   NOT reset the counter for the original step (per task: "different stepIds
   tracked separately").

9. **PrematureRunCompleted emits ONLY when `status === "passed"` AND there's an
   unresolved StepFailed in the last `REACT_PREMATURE_RUN_WINDOW (3)` events
   *preceding* the just-appended `RunFinished`.** "Unresolved" = no subsequent
   `StepCompleted` on the same stepId after the StepFailed (still within the
   plan.events list). This matches the canonical 4B premature-termination
   failure mode — agent emits passed RUN_COMPLETED while a step is still in the
   failed state. Failed runs are not gated (the failure is honest). If multiple
   unresolved failures exist, only the FIRST one (oldest) is named in the signal
   — keeps the signal payload bounded; the executor can re-scan the plan if it
   needs more.

10. **Reducer is `Effect`-returning with `never` error channel.** The Effect
    wrapper is for span/log instrumentation, not for failure propagation. All
    domain errors are converted to signals; all schema-decode errors are caught
    in-place. The `Schema.decodeUnknownEffect(AnalysisStep)` call is the only
    Effect-shaped operation; if it fails, we recover into a signal-emitting
    branch.

### Decisions made under ambiguity

1. **Skip-list for display updates after `agent_turn`.** The reducer adds the
   canonical ExecutionEvent (`AgentThinking` / `ToolCall` / `StepCompleted` /
   `StepFailed` / `RunFinished` / `PlanUpdate`) for each envelope. The
   subsequent display updates (`agent_thought_chunk`, `agent_message_chunk`,
   `tool_call`) carry the same semantic content and would double-count if also
   processed via `addEvent`. Skip those three. Keep `tool_call_update` (carries
   tool RESULTS — reducer doesn't have that data; `addEvent` translates it to
   `ToolResult` event). Keep `usage_update` (token telemetry, processed via
   `Stream.tap` separately, not `addEvent`). Keep all other types (no-op for
   `addEvent` anyway).

2. **First implementation used a sticky `inReactMode: boolean` that swallowed
   every display update once an agent_turn arrived. BMW v1 silently terminated
   because all abort messages (`[Reached maximum tool call rounds (15).
   Stopping.]`, `[Local agent: model returned empty content...]`) were eaten.
   Replaced with `expectsDisplaySkip: boolean` that resets per-envelope —
   skipping at most one display update following each agent_turn.** Local-agent
   abort paths (parse-failure, empty-content, MAX_TOOL_ROUNDS, doom-loop) emit
   an `agent_message_chunk` WITHOUT a preceding agent_turn for that round, so
   `expectsDisplaySkip` is false and the message flows through `addEvent`. T4's
   initial `executor-react-mode.test.ts` did NOT cover this — added the
   `preserves local-agent abort messages that arrive AFTER an agent_turn turn
   cycle (per-envelope skip rule)` test post-BMW to pin the behavior. Reviewer
   should scrutinize this correction explicitly.

3. **`as never` cast for the ACP SDK literal type union.** The SDK's
   `SessionUpdate` is a closed `type` union (not augmentable via interface
   merging). The JSON-RPC transport accepts arbitrary payloads at runtime.
   Three options considered: (a) cast `as never`, (b) declare module
   augmentation, (c) wrap in helper. (b) is impossible for type-alias unions
   in TypeScript. (c) just relocates the cast. (a) with a HACK comment is the
   minimum surface area. The receiver-side decode is type-safe via the
   `@neuve/shared` `AcpSessionUpdate` Schema.Union which DOES include
   `AcpAgentTurnUpdate`. Emission-side type-safety erased at one boundary; full
   decode-side safety preserved.

4. **Reducer must call `applyMarker` to update step.status.** Discovered via
   integration tests: legacy `addEvent` → `finalizeTextBlock` → `applyMarker`
   chain updated step.status; the reducer's direct event append did NOT.
   `runFinishedSatisfiesGate` falls through to `plan.allPlanStepsTerminal`
   which checks `step.status` of every step. Without `applyMarker`, all
   steps stayed at `"pending"` and the gate rejected legitimate `passed` runs.
   Fix is to call `applyMarker` immediately after appending the event in
   `handleStepDone` / `handleAssertionFailed`. The unit tests in T1 didn't
   catch this because they only asserted on events array, not on step.status —
   T4 should add explicit step.status assertions.

5. **`gemma-react` PlannerMode literal NOT added in R3.** The lead's brief
   suggested adding `gemma-react` to `PlannerMode` per the migration plan touch
   list. After analysis: the agent_turn wire works regardless of planner mode
   (template/none/oracle-plan/gemma-react are all compatible with
   gemma-runtime emitting agent_turn updates). Per PRD §R5, flipping the
   default to `gemma-react` lands in R5. T2's wire change is sufficient — no
   new mode literal needed in R3. Documented as deferred.

6. **HACK comment, not `Effect.die` for SDK type cast.** Per CLAUDE.md "No type
   casts (`as`) unless unavoidable" — adding `agent_turn` to a closed SDK type
   union without forking it IS unavoidable. Documented in code with a `HACK:`
   comment per CLAUDE.md "No comments unless it's a hack" exception.

## T3 — Adherence gate extension (`packages/supervisor/src/executor.ts`)

### Goal

Extend the existing `runFinishedSatisfiesGate` helper in `executor.ts` so that
a `RUN_COMPLETED=passed` is REJECTED when an `ASSERTION_FAILED` (`StepFailed`
event) appears in the last `REACT_PREMATURE_RUN_WINDOW (3)` events preceding
the `RunFinished` and is NOT followed by a matching `STepCompleted` on the
same `stepId`. This is the canonical 4B premature-termination failure mode
(Cemri 2025) — the model claims success while a step is unresolved.

### Files changed

| File | Kind | Purpose |
|------|------|---------|
| `packages/supervisor/src/executor.ts` | edit | Add private helper `hasUnresolvedAssertionInWindow(events, runFinishedIndex)` (pure function); rework `runFinishedSatisfiesGate` to find the RunFinished's index, walk back `REACT_PREMATURE_RUN_WINDOW` events, and reject the gate when a passed-status RunFinished follows an unresolved StepFailed. Existing rules preserved verbatim: missing RunFinished → reject; abort RunFinished → accept; otherwise fall back to `plan.allPlanStepsTerminal`. |
| `packages/supervisor/tests/executor-adherence-gate.test.ts` | edit | Add three new "R3 rule" tests: (a) passed RUN_COMPLETED with unresolved `ASSERTION_FAILED` in last 3 events → gate rejects; (b) passed RUN_COMPLETED with `ASSERTION_FAILED` followed by matching `STEP_DONE` → gate accepts; (c) failed RUN_COMPLETED with unresolved `ASSERTION_FAILED` → gate accepts (failure is honest). All four pre-existing Wave 1.B adherence tests still pass byte-identically. |

### Decisions made under ambiguity

1. **Inline the unresolved-window helper rather than reuse the reducer's
   identical-shape helper.** The reducer's `findUnresolvedFailureInWindow`
   in `react-reducer.ts` returns the offending `StepFailed` *or* `undefined`
   for signal payload purposes; the gate's `hasUnresolvedAssertionInWindow`
   returns `boolean`. Different return shapes; same loop body. I considered
   exporting the reducer's helper as a shared utility but rejected — the gate
   uses it on a plan whose RunFinished may not be at the very last index
   (events with text/thought chunks could in principle come after, though in
   practice `Stream.takeUntil(executed.hasRunFinished)` enforces last-index
   for RunFinished). Keeping them as separate small helpers preserves clarity
   of intent and lets each evolve independently. ~14 lines duplicated.

2. **Window boundary uses the RunFinished's index, not events.length.** The
   gate locates the RunFinished by walking events from the end, then uses
   `runFinishedIndex - REACT_PREMATURE_RUN_WINDOW` as the lower bound and
   `runFinishedIndex` as the exclusive upper bound. This isolates the gate
   from any future events that might arrive AFTER RunFinished — though the
   executor's `Stream.takeUntil` makes that case unlikely.

3. **Order of rule checks: missing → abort → R3-passed-with-unresolved →
   `allPlanStepsTerminal`.** The R3 rule slots in BEFORE the existing
   `allPlanStepsTerminal` check, not after. Reasoning: a passed run with
   `allPlanStepsTerminal === true` and an unresolved StepFailed (e.g., 2/3
   steps passed and step-3 failed but the agent emitted RUN_COMPLETED|passed)
   would have passed the existing gate. The new rule makes the failure visible.
   Failed-status runs do NOT take the new path, so honest failures still pass
   the gate (per existing semantics).

### Mid-T5 architectural fix #2 — per-envelope display skip

After Volvo run #2 (extNotification fix), I discovered a SECOND issue while
running BMW: my T2 `inReactMode: boolean` accumulator flag was set on FIRST
agent_turn and stayed `true` for the entire session, causing the supervisor
to swallow EVERY subsequent display update — including the local-agent's
crucial wrapper messages like `[Local agent: model returned empty content
at round 1...]` and `[Reached maximum tool call rounds (15). Stopping.]`.

The local-agent emits these abort messages via `agent_message_chunk` WITHOUT
a preceding `agent_turn` for that round (because the abort happens AFTER the
chat call but BEFORE successful envelope parsing). With `inReactMode=true`
sticky, all such messages were filtered out, leaving the supervisor blind
to terminations.

**Fix**: switched `inReactMode: boolean` → `expectsDisplaySkip: boolean`.
The flag is set to `true` ONLY immediately after an `agent_turn` is
processed; on the very next session update, if it's a display type,
SKIP it AND set `expectsDisplaySkip=false`. On any other update kind
(tool_call_update, usage_update, abort messages), `expectsDisplaySkip`
resets to false. This means abort messages flow through `addEvent`
unchanged.

Files changed: `packages/supervisor/src/executor.ts` (renamed accum field,
replaced sticky check with per-envelope reset). All 119 supervisor tests
still green — none of them tested the abort-message-after-react-turn flow
