## T1 — `packages/evals/src/runners/gemini.ts` (Gemini-react ReAct runner)

### Goal

Build the A:B baseline counterpart of the gemma-react production runner. The
runner drives the same supervisor pipeline (`runRealTask` → `Executor.execute`
→ `reduceAgentTurn` → adherence gate) but with Gemini Flash 3 as the in-process

### Decisions made under ambiguity

1. **Use `jsonSchema()` from `ai` + Effect Schema for response constraint**, not
   a Zod mirror. The AI SDK supports both Zod and JSON Schema 7 via `jsonSchema()`
   wrapping. The Effect `Schema.toJsonSchemaDocument(AgentTurn)` already produces
   a Draft 2020-12 document we use to constrain Ollama's `format` parameter
   (R2's tool-loop). Using the same schema source for Gemini eliminates the
   "Zod and Effect Schema diverge" failure mode and respects DRY. The
   `validate` callback re-runs `Schema.decodeUnknownExit(AgentTurn)` for
   defense-in-depth — if Gemini's responseSchema implementation accepts a
   value that the Effect Schema would reject (e.g. extra properties), we
   surface it as a typed failure before `parseAgentTurn` runs in the loop.

2. **In-process Agent layer (Layer.effect + Layer.succeed) instead of a
   subprocess.** The local-agent path spawns a child process because (a) it
   needs to run in production, (b) it isolates Ollama interaction, (c) it
   matches the ACP wire. None of those reasons apply to the gemini-react
   eval runner — Gemini calls go directly via `@ai-sdk/google` from the eval
   process. The in-process layer is ~150 lines vs. the 600+ lines of
   `local-agent/src/agent.ts` + `acp-client.ts`. Lifecycle is `Stream.callback`
   which opens a managed scope per `stream()` call; the McpBridge acquires
   under that scope and releases on stream completion (Stream.takeUntil
   triggers the cleanup when RUN_COMPLETED arrives or the executor's grace
   period fires).

3. **Reuse `@neuve/local-agent/mcp-bridge`** instead of duplicating the
   browser-mcp + flatten-oneOf + wrapper-key auto-wrap logic. Exposed
   `./mcp-bridge` as a public export of `@neuve/local-agent` (along with
   `./ollama-client` for symmetry). No type leakage — `OllamaToolDefinition`
   in the bridge is structurally the OpenAI tool function shape, not
   Ollama-specific. The mild side effect: importing the bridge transitively
   imports `log.ts` which creates `.perf-agent/local-agent.log` at module
   load. Acceptable for eval runs (the directory is gitignored; co-locating
   gemini-react and gemma-react logs is actually convenient for A:B
   debugging).

4. **`Stream.callback` over `Stream.unwrap(Effect.gen(...))`.** The first
   typecheck pass tried `Stream.unwrap` and hit "Stream.unwrapScoped not
   exported" (it isn't in Effect 4) plus Queue.end / Cause.Done plumbing.
   `Stream.callback` is the v4 idiom for "build a stream from a queue
   filled by a scoped Effect" — it manages scope, queue lifetime, and
   error propagation in one primitive. The loop becomes a plain emit
   callback (`emit: (update) => void`) without queue/scope plumbing.

5. **`plannerMode: "none"` for gemini-react default**, not "template" or
   "oracle-plan". The ReAct loop owns plan authorship via PLAN_UPDATE
   envelopes; pre-decomposing would short-circuit the very capability T5's
   regression sweep needs to measure. T3 adds `"gemma-react"` as a 4th
   PlannerMode literal; we'll switch the default in the same wave (per
   PRD §R5 line 276 — `gemma.ts` flips, gemini.ts implicitly tracks).

6. **Doom-loop and max-rounds thresholds copied verbatim from
   `@neuve/local-agent/tool-loop.ts`** (3 and 15 respectively). Different
   thresholds would make the A:B comparison non-apples-to-apples: a model
   that gets 5 rounds vs. 15 rounds is structurally different from one
   that gets fewer USEFUL turns inside the same budget. T5's regression
   table needs the ratio "rounds-used / rounds-budget" to mean the same
   thing for both runners.

7. **Token telemetry via `usage_update` Acp event**, mirroring the
   local-agent's wire pattern. The supervisor's executor (R3+R4) already
   handles `usage_update` → `TokenUsageBus.publish` — we don't need to
   touch the supervisor. Each generateObject call emits `_meta:
   { promptTokens, completionTokens, totalTokens }` so the trace recorder
   writes one `token_usage` event per round and the budget monitor (R4)
   sees Gemini's prompt growth. Gemini's `usage.inputTokens / outputTokens`
   are flat numbers (not nested) per AI SDK 7's `LanguageModelUsage` type.

8. **`GeminiReactCallError` is a typed Effect error**, converted to
   `AcpStreamError` at the agent-layer boundary so the supervisor's
   `ExecutionError` wrapper still matches and the rest of the pipeline
   doesn't need to know about Gemini-specific errors. The conversion is a
   single `Effect.catchTag("GeminiReactCallError", error => new
   AcpStreamError({...}).asEffect())` — no broad catchAll/mapError.

## T2 — `packages/evals/src/runners/gemma-oracle-plan.ts` (debug ablation runner)

### Goal

Build the third leg of PRD §4 Decision #8's eval-runner triad: a Gemma-react
runner that uses a Gemini-decomposed oracle plan instead of letting Gemma
author the plan via PLAN_UPDATE. This isolates planning quality from
execution quality:

- **gemma-react** (T3 default): Gemma owns plan + execution.
- **gemini-react** (T1): Gemini owns plan + execution (frontier baseline).

**planning**; distillation should focus on plan authorship (PRD §9
Decision #9 mentions PLAN_UPDATE quality as a distillation target). If the
two tie, the gap is **execution**; distillation focuses on tool selection
and observation interpretation.

### Decisions made under ambiguity

1. **`"gemma-react"` and `"none"` are functionally equivalent in the
   harness** — both skip pre-decomposition. They're kept distinct because
   the literals carry different semantic intent: `"none"` is the legacy
   "I don't want any plan" escape hatch; `"gemma-react"` is the deliberate
   "ReAct mode, agent owns plan via PLAN_UPDATE" signal. Future tooling
   (eval column headers, log filters) can branch on the literal.
2. **Eval files keep their current defaults**, only the validation tuple
   widens. `wave-r3-react-replay` defaults to `"template"` to preserve
   the regression-baseline behavior the eval was designed to test;
   flipping its default would change what the eval measures. The new
   `"gemma-react"` literal becomes available via env var override.
3. **No `Schema.import("PlannerMode")` refactor in the eval files.**
   Importing the canonical schema from `errors.ts` would be DRY but adds
   a wider blast radius across 5 eval files. Adding the literal one place
   per file is a lower-risk diff. Future cleanup could DRY this if
   PlannerMode keeps growing.
4. **Defect on `decompose(mode="gemma-react")` rather than no-op**, even
   though the supervisor's `runRealTask` short-circuits before reaching
   it. The defect is a contract guard — if a future caller (or a
   regression in `runRealTask`) reaches the decomposer with `gemma-react`,
   we want it loud, not silent. Matches the existing `"none"` behavior.

### Decisions made under ambiguity

1. **Render plan_update as the wire-canonical AgentTurn JSON envelope**,
   not as a status-marker-style line (`PLAN_UPDATE ${JSON.stringify(...)}`).
   The distillation target is "fine-tune Gemma to emit AgentTurn JSON
   envelopes that the supervisor's reducer consumes." Training on the
   exact wire format is what closes the distribution gap. The existing
   status_marker rendering uses a different format because those events
   pre-date the ReAct migration and live in the older
   pipe-delimited-marker world; refactoring those is out of scope for T4.
   New events follow the new format.
2. **Redact `payload` recursively via the existing `redactSensitiveKeys`
   path.** PLAN_UPDATE payloads carry `AnalysisStep` objects which
   shouldn't contain secrets, but the same teacher-data filter rules
   apply across all envelope fields. Keeps the redaction surface
   consistent.
3. **`stream_terminated`, `token_usage`, `task_tokenomics` events are
   skipped**, same as before. These are harness-level signals, not
   learnable model output. Updated the trailing comment to mention all
   three so future readers don't wonder why each one is missing.
4. **`payload` is `Schema.optional(Schema.Unknown)`** rather than
   `Schema.Unknown` directly. PlanUpdate.action="remove" has no payload;
   marking it optional honors that contract and keeps the event compact
   on disk.
5. **Trace event type name is `plan_update`** matching PRD §R5 line 278
   verbatim. Future replay tools and analysis scripts pin this literal.

## T5 — Full 20-task A:B regression sweep + report

### Goal

Build the eval driver + post-run aggregation script that produces the
headline regression report comparing the 3 runners (gemma-react vs
gemini-react vs gemma-oracle-plan) across the full 20-task fixture set.
Per PRD §R5 line 279 + §6.3, this is the Phase R5 exit criterion: full
20-task run for all three runners, report committed to
`docs/handover/harness-evals/baselines/`.

### Realistic scope for this engineering session

A live 60-evaluation sweep (3 runners × 20 tasks) takes hours of
wall-clock — each task can spend 10-15 minutes of headless Chromium +
generateObject latency + Ollama latency. That's not safely runnable
inside one engineering session. Per the lead's brief: "Day 3-4: T5
(20-task A:B sweep). Long-running."

**Engineer deliverable**: ship the eval driver, scoring/aggregation
helpers, report generator, and unit tests so the lead can kick off the
sweep with one command. The actual sweep numbers land in the report
when the lead runs `pnpm eval:wave-r5-ab` + `pnpm wave-r5-ab:report`.

This is the same pattern Wave 4.5 followed (per
`project_baseline_eval_strategy.md` — `B1` and `B2` baselines lived in
`docs/handover/harness-evals/baselines/` and were generated post-hoc).

### Audit-2 — T4 rollTrajectory option in teacher-data exporter

**Brief excerpt (T4):** "Apply R4's `rollTrajectory` to keep example
prompt sizes under context budget."

My initial T4 implementation added the `plan_update` event mapping but
did NOT apply rollTrajectory to per-turn samples. Per-trajectory samples
are full traces by definition — no rolling needed there. But per-turn
samples accumulate prompt context across turns, so by turn 14 the
prompt is 31 messages long; without rolling, training-time prompt
sizes can exceed the 96K/120K runtime budget.

**Fix:**
- `packages/evals/src/distill/types.ts`: added `rollTrajectory?: boolean`
  to `ExportOptions` (default `false`, preserves existing behavior so
  pre-R5 distillation runs are unchanged).
- `packages/evals/src/distill/teacher-data-exporter.ts`: new
  `applyTrajectoryRollToSample` helper. Remaps `role: "tool" → "user"`
  before rolling because `partitionTrajectory` only matches `role: "user"`
  observations (the exporter emits tool results as the OpenAI-standard
  `tool` role; structurally they ARE observations and the remapping is
  semantically correct). The TARGET assistant message (last in the
  sample) is preserved verbatim — that's what the model trains to emit;
  rolling its content would corrupt the learning target.
- Applied only to per-turn granularity (per-trajectory keeps full traces).
- New test in `packages/evals/tests/teacher-data-exporter.test.ts`:
  builds a 14-turn synthetic trace, asserts the rolled per-turn sample
  has fewer messages than the unrolled, contains `<trajectory_summary>`,
  and preserves the target assistant verbatim.

**Decisions made under ambiguity (post-audit):**

1. **`rollTrajectory: false` as default** — the existing distillation
   pipeline (`pnpm distill:export`) consumes traces from prior waves;
   silently changing the default would alter teacher-data shapes
   without user opt-in. Future distillation runs targeting the new
   `browsing-gemma` LoRA can opt into rolling via the option.
2. **Tool-role remapping over altering rollTrajectory** — touching
   `@neuve/shared/trajectory.ts` would affect the live runtime
   tool-loop's rolling behavior, which has been validated with the R4
   integration tests. Keeping the trajectory module unchanged and
   doing the role remap in the exporter is lower-risk.
3. **Drop tool-call structure on older turns** — when the summarizer
   produces `<event>` lines for older turns, tool-call shape on those
   older assistant messages is lost. That's correct for distillation:
   each older turn already produced its OWN per-turn sample where its
   tool-call structure was the training target. In the late-turn
   sample, those older turns are PROMPT context, and the summary line
   is sufficient context (tool name + observation outcome) without
   re-paying training capacity for shape that's already learned.

### M1 — Promise-level error swallow on McpBridge close

**Defect:** `gemini-agent.ts` had
`bridge.close().catch(() => { /* swallow */ })` — canonical
CLAUDE.md "Never Swallow Errors" ban. McpBridge spawns a stdio child
via `process.execPath`; close hangs would silently produce zombies
during a 60-eval sweep.

**Fix:** Replaced with `Effect.tryPromise` wrapping `bridge.close()`,
piped through `Effect.catchTag("AcpStreamError", error =>

during a 60-eval sweep we want the next task to start cleanly even if
one runner's bridge had a hung close. Two reasons for warning-not-die:
(a) close is best-effort during teardown, (b) crashing teardown would
abort the entire sweep on a recoverable issue, defeating the long-run
robustness purpose of the sweep.

### m1 — JSONSchema7 double cast HACK comment

`gemini-react-loop.ts:64` used `as unknown as JSONSchema7`. The cast
is genuinely unavoidable: Effect's `Schema.toJsonSchemaDocument`
returns a `JsonSchema.Document<"draft-2020-12">`; AI SDK's
`jsonSchema` helper expects `JSONSchema7` (draft-07). The two specs
share most fields but the meta-keys differ (`$defs` vs `definitions`),
and TypeScript's structural typing can't bridge them without
`as unknown`. Documented with a `// HACK:` block citing both specs
and noting that AI SDK 7's Gemini provider tolerates `$defs` at
runtime.

const UnknownJsonShape = Schema.fromJsonString(Schema.Unknown);
