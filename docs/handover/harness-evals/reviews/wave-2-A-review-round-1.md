# Review: Wave 2.A — First-class interaction tools (Round 1)

## Verdict: REQUEST_CHANGES

The five tool wrappers, service interfaces, errors, constants, mock tests, and MCP-layer registration module are all present, formatted, typecheck-clean, and the 28-test suite passes deterministically twice. Effect v4 patterns are mostly honored — `ServiceMap.Service`, `Schema.ErrorClass`, `Effect.fn`, no banned error combinators. But the wiring, the production-live layer, and the default production behavior have serious defects: the MCP tools are **never actually registered with the MCP server**, the live `NetworkIdleSampler` parses a format chrome-devtools-mcp never emits (so `waitForNetworkIdle` is a silent no-op in production), the live `RefResolver` uses substring matching that will false-positive and false-negative on realistic uids, the live `select` handler is wired to the `fill` tool, and the test suite never exercises the live layer at all. This is a textbook production-vs-test divergence pattern per `feedback_no_test_only_injection_seams.md`.

### Findings

- **[CRITICAL] The five new MCP tools are defined but never registered with the MCP server.** (`packages/browser/src/mcp/server.ts:93-95`, `packages/browser/src/mcp/tools/interactions.ts:111`). `registerInteractionTools` is exported from `interactions.ts` but `server.ts` only calls `registerInteractTool`, `registerObserveTool`, `registerTraceTool`. I grep-verified: `registerInteractionTools` has exactly one hit repo-wide — its own declaration. The diary at line 31 claims `server.ts` was "updated to wire services" — this is false. Net effect: the whole user-visible Wave 2.A surface is dead code from the agent's perspective. The new service layers in `runtime.ts` satisfy the runtime's R-channel but are consumed by nothing. DoD line "agent can click without evaluate_script" is not demonstrable end-to-end because the agent literally cannot call `click` via MCP.

- **[CRITICAL] Live `NetworkIdleSampler.inFlightCount` parses a string format chrome-devtools-mcp never emits, so `waitForNetworkIdle` is a no-op in production.** (`packages/browser/src/tools/live.ts:30-36, 119-128`). The regex is `/(\d+)\s+(?:in[- ]?flight|pending|active)/i`. The actual `list_network_requests` output format (per verbatim captures in `.specs/observability-output-format.md:96-123`) is `## Network requests\nShowing N-M of T (Page …).\nreqid=<n> <METHOD> <URL> [<status|pending>]`. There is no leading integer followed by the words "in-flight"/"pending"/"active" anywhere in that output. I reproduced against the spec fixture: regex → `null` even when the text contains two `[pending]` lines, so `parseNetworkInFlight` returns 0. `Effect.repeat(..., until: (idle) => idle)` then exits immediately because the first sample is already "idle". Every `click`/`fill`/`hover`/`select` will appear to wait for network idle when in reality it returns in microseconds. The unit tests hide this by stubbing `NetworkIdleSampler` entirely — the live path is never exercised. This is the same production-vs-test divergence class flagged by memory `feedback_no_test_only_injection_seams.md`. Either subscribe to CDP network events (proper fix) or at minimum count occurrences of `\[pending\]` per line with a sentinel-anchored regex.

- **[CRITICAL] Live `RefResolver.resolveRef` uses substring matching, so it will false-positive and false-negative on realistic uids.** (`packages/browser/src/tools/live.ts:103` — `if (!text.includes(ref))`). The chrome-devtools-mcp accessibility snapshot contains many numbers — node ids, ARIA levels, list indices, URL digits. A ref of `"3"` matches any snapshot with a `3` character anywhere. Conversely a uid like `"node_42"` might be matched or not depending on whether the snapshot ever renders the literal string. The test fixture uses opaque names (`"dropdown-1"`, `"input-7"`, `"menu-2"`) which sidesteps the issue, but real chrome-devtools-mcp uids and SOM numeric refs will both collide. The resolver is not a no-op (good), but it is not a correct resolver either. `isVisible` (same file, line 84) has the same defect. If the decision is "pass-through to chrome-devtools-mcp uids, don't validate on our side," then don't validate at all — just build the handle. The half-validation is worse than no validation because it fails opaquely on perfectly valid refs.

- **[CRITICAL] Live `ElementHandle.select` calls the chrome-devtools-mcp `fill` tool, not `select`.** (`packages/browser/src/tools/live.ts:75-81`). `devtools.callTool("fill", { uid: ref, value: String(option) })`. chrome-devtools-mcp exposes both `fill` and `select_option` (or equivalent) — `fill` types text into inputs, it does not select an `<option>` from a `<select>`. Behavior will be at best "no-op" (HTML `<select>` elements ignore keyboard fill text), at worst it will set `.value` to a string that doesn't match any option and leave the select in a weird state. Even if the chrome-devtools-mcp `fill` happens to accept `<select>` elements, a numeric `option: 2` gets stringified to `"2"` and set as the value — not used as an index into options[]. The tool's semantics (`option: string | number` where number is "by index") are not honored at all. The test layer's `select` hook records the arg but never calls the live layer, so this bug is invisible to tests.

- **[MAJOR] Services swallow `DevToolsToolError` silently in two layers, hiding all CDP failures from the caller.** (`packages/browser/src/tools/live.ts:123-126, 141-144`). `networkIdleSamplerLayer` and `snapshotTakerLayer` both do `Effect.catchTag("DevToolsToolError", () => Effect.succeed({ content: [] }))`. This is the exact "never swallow errors" banned pattern from CLAUDE.md. If chrome-devtools-mcp is wedged or returns an error while we're sampling idle/taking a snapshot, we lie and return "idle / empty snapshot." The snapshot lie in particular is damaging: the tool then returns an empty-string `ToolResult.snapshot.text` to the agent, which the agent will read as "page is empty / action did nothing" — hiding a real infra failure as a semantic one. Should be `Effect.catchTag("DevToolsToolError", Effect.die)` or surface as a domain-specific error.

- **[MAJOR] Six `as CallToolContent` type casts in the live layer are avoidable.** (`packages/browser/src/tools/live.ts:84, 102, 124, 127, 142, 146, 181`). CLAUDE.md: "No type casts (`as`) unless unavoidable." The diary explicitly acknowledges the casts and tries to narrow them, but they are not unavoidable: `DevToolsClient.callTool` / `takeSnapshot` / `listNetworkRequests` return a structurally-equivalent `CallToolResult` type already declared in `devtools-client.ts:6-14`. Either import that type (after exporting it) or delete the duplicate `CallToolContent` interface and type `extractText` / parse helpers against the real return type. Fix: export `CallToolResult` from `devtools-client.ts` and use it directly in `live.ts`.

- **[MAJOR] Two `as` casts in `interactions.ts` are also avoidable.** (`packages/browser/src/mcp/tools/interactions.ts:49, 51, 218`). Line 218: `(parsed.state ?? "visible") as WaitForState` — `WaitForInput.state` is already a `z.enum(WAIT_FOR_STATES).optional()` where `WAIT_FOR_STATES = [...] as const satisfies WaitForState[]` could constrain the type without cast. Lines 49/51: the `Cause.squash` message extraction should use a schema/type-guard helper rather than two object-shape casts.

- **[MAJOR] Default production `RefResolver` is the pass-through layer, which is not an explicitly-temporary seam for Wave 2.C to override.** (`packages/browser/src/mcp/runtime.ts:11-23`, `packages/browser/src/tools/live.ts:38`). The production `McpRuntime` ships with `refResolverLayer` as THE resolver. There is no `// HACK:` comment, no named Wave-2.C handover symbol (e.g. `defaultRefResolverLayer` vs. a future `somRefResolverLayer`), and no single layer-composition seam documented for 2.C to swap. When 2.C lands its SOM-backed resolver, the engineer will either have to `Layer.provide(somLayer)` late (overriding `refResolverLayer` via Effect's later-wins semantics, which works but is implicit) or rewrite `runtime.ts`. More concerning: the pass-through resolver is untested against a real page in this wave (all tests mock), so we will not know whether SOM-numbered refs can cohabit with raw chrome-devtools-mcp uids when 2.C arrives. This is precisely the fallback-with-default anti-pattern from memory `feedback_no_test_only_injection_seams.md`.

- **[MAJOR] No integration or end-to-end test exercises the live layer.** (`packages/browser/tests/tools/*`). Every test uses `makeTestLayers` which replaces all four services. The live `parseNetworkInFlight` regex, substring ref resolver, select-is-fill bug, and silent-swallow catches are all invisible to CI. Effect-TS lets us at minimum write a "contract" test that boots the live layers with a fake `DevToolsClient` stub returning the verbatim fixtures from `.specs/observability-output-format.md`. Without that, the test suite's 28-pass result is misleading — it verifies the generic Effect plumbing only.

- **[MAJOR] `wait_for` with `state: "hidden" | "detached"` inverts the wait semantics but does not distinguish "not attached yet" from "was attached then removed."** (`packages/browser/src/tools/live.ts:166-177`). The `desired = state !== "hidden" && state !== "detached"` toggle treats all four states as two binary modes. A test asking for `detached` on a never-present element returns success immediately — arguably correct for "detached," wrong for "hidden" on an element the site hasn't rendered yet because it conflates "never existed" with "existed then left." At least flag this in a comment or carry a distinct probe for "existed at least once."

- **[MINOR] `captureSnapshot` / `inFlightCount` inner fns omit `yield* Effect.logDebug` at the call site for observability, despite the CLAUDE.md "Log high-frequency reads at Debug level" rule.** (`packages/browser/src/tools/helpers.ts:24-29`). There's a `snapshotSize` annotation but no debug line. Each tool does log once at Info, fine — but the per-call helpers are the right level for Debug.

- **[MINOR] Tool tests use `vitest`'s raw `it` + `Effect.runPromise` / `runPromiseExit` instead of `@effect/vitest`'s `it.effect`.** (`packages/browser/tests/tools/click.test.ts:1-11`, applies to all five test files). CLAUDE.md says "Use `@effect/vitest` with `it.effect`". The current tests work, but they miss the automatic Scope provision and Cause formatting that `it.effect` gives. Not a blocker on its own, but combined with the "no live-layer test" gap it compounds the review concern.

- **[MINOR] `ElementHandle.isVisible` is dead code in the default pipeline.** (`packages/browser/src/tools/types.ts:59`, `live.ts:82`). The comment in the diary says `WaitForEngine.waitForRef` uses it as a cheap probe — but the live `waitForEngineLayer` (`live.ts:179-185`) builds its own `refProbe` via `devtools.takeSnapshot(...).includes(ref)` and never touches `ElementHandle.isVisible`. If the contract is "resolve-then-probe," either call `element.isVisible()` or drop the method from `ElementHandle`.

- **[MINOR] `decodeRef = Schema.decodeSync(ToolRef)` throws synchronously on failure — not wrapped in Effect, so the MCP request handler will surface an uncaught exception to the transport instead of a typed MCP error.** (`packages/browser/src/mcp/tools/interactions.ts:71, 131, 156, 176, 194, 218` via `resolveWaitForTarget`). `ToolRef` is a brand over `Schema.String`, so a non-string ref would throw. In practice Zod has already rejected non-strings at `ClickInput.parse(input)`, so the throw path is dead, but the style hint in CLAUDE.md says "Use `Effect.try` for sync" — wrapping `decodeSync` in an Effect would be more consistent.

- **[INFO] Good Effect hygiene on the services themselves: `ServiceMap.Service<T>()("@devtools/...", {...})` with `static layer` or `Layer.effect`, typed errors via `Schema.ErrorClass` with `_tag` + class-field `message`, branded `ToolRef`, `Effect.fn` with span names, `Effect.annotateCurrentSpan` at the right spots, no `null`/`Effect.option`/`Effect.ignore`/`catchAll`/`mapError`/`orElseSucceed` in the 2.A source.** Also no barrel `index.ts` in `tools/`. kebab-case file names. These are the parts that passed.

- **[INFO] Diary deviations 1–5 all truthful with one exception:** deviation #3 ("default RefResolver treats ref as uid pass-through") understates the risk: the pass-through is not a clean handover seam, is untested, and overlaps with a broken substring match. The Wave 2.C reviewer needs to know the seam is implicit, not explicit.

### Suggestions (non-blocking)

- Export `CallToolResult` from `devtools-client.ts` and adopt it across `live.ts` to remove the six `as` casts.
- Move the default production `refResolverLayer` name to something like `uidPassThroughRefResolverLayer` and document at the top of `live.ts` (with a `// HACK:` comment) that this is scaffolding for Wave 2.C to override.
- Add a contract test that provides a fake `DevToolsClient.layer` returning verbatim fixtures from `.specs/observability-output-format.md` — exercise `NetworkIdleSampler.inFlightCount`, `RefResolver.resolveRef`, `WaitForEngine.waitForRef` against realistic output. This alone would have caught all three CRITICAL findings.
- Swap the tests to `@effect/vitest`'s `it.effect` — you can drop `.pipe(Effect.provide(layer))` in favor of top-level provision, which reads cleaner.
- When 2.C lands, make the layer swap an explicit named seam: `export const refResolverLayerLive = somLayer.pipe(Layer.provideFallback(uidPassThroughLayer))` so it's obvious where the override happens.

---

Round 1 verdict stands. Staying alive for Round 2 once the engineer addresses the four Criticals and the Major cluster.
