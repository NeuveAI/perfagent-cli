## Methodology

For each probe:
1. Load production system prompt + production MCP tool definitions (from
   `docs/handover/q9-tool-call-gap/probes/browser-mcp-tools.json`).
2. Reconstruct conversation history through the failing turn using the
   trace ndjson as ground truth.
3. POST to Ollama `/api/chat` (probe 1, 2) or LM Studio `/v1/chat/completions`
   (probe 3) with `num_ctx=131072`, `temperature=0.1`, identical message array.
4. Vary the request shape across modes:
   - `full`: `tools` + `format` (production shape)
   - `no-tools`: only `format`
   - `no-format`: only `tools`
   - `bare`: neither
5. Repeat REPS times per mode; capture content length, eval count, prompt
   eval count, finish reason, elapsed.

Real-eval validation: also re-ran the actual eval harness for two
known-empty-content tasks (`journey-9-form-wizard` and
`calibration-2-single-nav-news`) via `EVAL_TASK_FILTER` + `EVAL_R5_SKIP_RUNNERS`
to ensure the bug isn't trivially gone for unrelated reasons.

### Probe 3 — LM Studio `llama.cpp` runtime (`gemma-4-e4b` GGUF Q4_K_M)

Same conversation, same parameters, against `localhost:1234/v1/chat/completions`
with `response_format: { type: "json_schema", json_schema: ... }` instead of
Ollama's `format`.

| Mode | Reps | Result |
|---|---|---|
| `full` (tools + response_format) | 5 | **HTTP 400 every time:** `{"error":"Cannot combine structured output constraints with lazy grammar"}` |
| `no-tools` (response_format only) | 3 | 0 empty; valid envelopes |
| `no-format` (tools only) | 3 | 0 empty; valid envelopes |
| `bare` | 3 | 0 empty; valid envelopes |

**The LM Studio llama.cpp runtime EXPLICITLY REJECTS combining `tools`
with `response_format: json_schema`.** The error message names the exact
incompatibility: a "lazy grammar" (the conditional grammar that activates
when the model decides to invoke a tool) cannot coexist with a "structured
output constraint" (the always-on JSON-Schema grammar). Both apply to the
same logits-masking layer in llama.cpp; LM Studio refuses the request rather
than letting the grammars collide.

Ollama 0.22.0 accepts the same combination (probe-2 full mode succeeds 30/30
on our reconstruction) but LM Studio's hard rejection tells us the
combination is architecturally suspect upstream — Ollama is doing something
non-standard to make it work, and our intermittent empty-content events on
production sweeps fit a "grammar collision under specific conditions" model.

## P3 recommendation

**Drop the `tools` field from the Ollama `/api/chat` request when `format` is
provided.** One-line change in
`packages/local-agent/src/ollama-client.ts` `buildRequestBody` — strip the
`tools` block (or strip it conditionally on `options.format !== undefined`).

Why this is safe:
1. `tool-loop.ts:49` — comment: "native `message.tool_calls` are intentionally
   ignored". The loop dispatches on the JSON envelope's `_tag` discriminator,
   not on Ollama-emitted `tool_calls`.
2. `tool-loop.ts:189` — comment: "The format-grammar guarantees the assistant
   message IS an AgentTurn JSON envelope".
3. Probe-2 `no-tools` mode (format only) produces correct `THOUGHT`/`ACTION`
   envelopes — Gemma still selects the right tool name from the system
   prompt's `<tool_catalog>` prose.
4. LM Studio llama.cpp's hard rejection confirms `tools + response_format` is
   architecturally unsupported upstream; aligning Ollama with that contract
   removes the latent grammar-collision pathway.

Why it's preferred over a harness safety net:
- The safety net (retry on empty-content with grammar relaxed) papers over
  the architectural smell. Per `feedback_no_test_only_injection_seams.md`,
  we should fix the root cause not retry around it.
- LM Studio's error message **names** the bug class. We have an explicit
  upstream signal to align with.
- Removing `tools` is a smaller change (one line vs new retry state machine
  + telemetry + live integration tests).

Why not migrate to LM Studio + MLX:
- MLX gemma4:e4b GGUF/MLX variant doesn't exist on the Hub yet.
- LM Studio llama.cpp would still refuse the production request shape, so we
  must drop `tools` regardless.
- Migration is heavier than necessary for the demonstrated fix.

### New failure mode surfaced — schema-invalid envelopes

Side effect of the fix: 10/20 tasks now bail on `parseAgentTurnFromString` SchemaError ("non-schema-valid agent output") at round 3-5. R7 phase-7 had 1/20 of these. The pattern across all 10 traces is the same — Gemma emits commands that don't exist in the strict per-tool union:

- `interact { command: "click", selector: "..." }` — `click` isn't a recognized `interact` subcommand (the flat `click` tool is separate).
- `interact { command: "type", selector: "...", text: "..." }` — `type` isn't a recognized subcommand (canonical name is `fill`).
- `interact { command: "title", ... }` — pure hallucination.
- `observe { command: "click", selector: "..." }` — wrong tool entirely.

Diagnosis: pre-fix, the `tools` field gave Gemma a structured schema for each tool's commands (the canonical+shorthand union per dispatcher). Post-fix, Gemma relies only on the system prompt's `<tool_catalog>` prose (`buildLocalAgentSystemPrompt` lines 127-131), which is sparse AND inconsistent with the strict union. Specifically the prompt says: `"interact — navigate, click, type, fill. command='navigate' with url is the entry path."` — listing `click`, `type`, `fill` as interact commands when the strict per-tool union expects `click` to be a top-level tool and `fill` to be the canonical interact-fill verb (no `type` alias).

The format grammar (`AgentTurnLoose`) is permissive on `args: Schema.Unknown`, so the model can emit any args shape and the grammar lets it through. The strict runtime parse rejects.

Aggregate impact: +3 hard-fail tasks vs R7 phase-7 BUT mean step-coverage went UP (0.398 → 0.423) because previously-empty-content tasks now run far enough to reach more key nodes before bailing. So in stepwise progress terms the fix is a net win, but in pass/fail terms it surfaces a previously-masked prompt/schema mismatch.

### Surfaced for lead/reviewer attention

The schema-invalid regression isn't in the R8 P3 scope but should be queued for follow-up. Options:

1. **Tighten the system prompt** — replace `interact — navigate, click, type, fill` with the canonical union (e.g. `interact — supports navigate(url), click(uid), fill(uid, value), hover(uid)…`) so prose matches the strict schema.
2. **Loosen the strict per-tool union** — accept `type` as alias for `fill`, fold the flat `click`/`fill`/`hover`/`select` tools into the `interact` union so there's only one way to express any operation.
3. **Restore `tools` field with a stripped schema** — pass tools but with `parameters: { type: "object", additionalProperties: true }` so the lazy-grammar collision is avoided (the schema is empty enough not to constrain) but the model still sees the tool names. Untested whether Ollama's tool-call grammar fires on an empty schema.

Lead's call which to pursue. Not in scope for R8 / task #3 — surfacing as a follow-up.
