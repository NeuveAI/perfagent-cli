### Prompt-design principle (worth keeping for future edits)

**Density + proximity beats section-organized clarity for small-model literal-readers.**

V1's effective form was a single dense line packing alias-warning + per-verb-shape together:
```
args fields are exact (no `selector`/`ref`/`text` aliases): navigate{url|direction}; click/hover{uid}; fill{uid,value}; type{text}(focused-only); ...
```
→ 0/20 selector failures.

V2 reorganized for "discrimination signal" (per-dispatcher purpose, per-verb shape on its own indented line):
```
Args fields are exact — no `selector`/`ref`/`text` aliases. uid is from observe.snapshot.
  navigate{url}  (or {direction})
  click{uid}; hover{uid}
  fill{uid,value} — for form-field inputs (after observe.snapshot).
  type{text} — types into focused element ONLY ...
```
→ 12/20 selector failures.

The reorganization was structurally cleaner for human readability but Gemma 4 E4B reads section-by-section and over-trusts the first authoritative-feeling block. The 4-line gap between alias-warning and per-verb shapes was enough that gemma constructed `interact{command:"click", selector:...}` from the dispatcher block (which prominently lists `click` as an interact verb) and never re-engaged the args section.

Symptom signature: mean turns dropped 10.3 → 6.3, mean tokens 90k → 21k, schema-invalid rose 7 → 14. Faster bailing on the same wrong shape = the prompt's denser top "answers" planning quickly without forcing re-reads of subordinate sections.

Two related model-as-literal-reader traps observed in this wave:
- v2 micro-trap: `(focused-only)` parenthetical literal-copied as a field name `focused-only:true` (j-7 in v1 sweep). Avoid kwarg-shaped parentheticals near field-shape examples.
- v2 macro-trap: structural section-organization broke inline proximity. Avoid splitting a constraint and its examples across multiple lines if the small model relies on local context.

Forward heuristic for prompt edits targeting small models:
1. Keep constraints + their satisfaction examples on the **same line** when budget permits.
2. Prefer prose-folded over parenthetical-annotated near field-shape mentions.
3. If you must reorganize, sweep before/after — section-organized clarity does NOT free.

### Three takeaways for future work

1. **Prompt-design principle (density + proximity)**: for small-model literal-readers like gemma 4 E4B, packing constraints + their satisfaction examples on the SAME line beats section-organized clarity. Splitting a constraint and its examples across multiple lines lets the model over-trust the first authoritative-feeling block and never re-engage subordinate sections. Symptom signature: faster bailing on the same wrong shape (mean turns ↓, mean tokens ↓, schema-invalid ↑).

2. **Literal-reader trap (micro)**: avoid kwarg-shaped parentheticals near field-shape examples — gemma copied my v2 `(focused-only)` annotation as a literal field name `focused-only:true` (j-7). Plain-English parentheticals are lower risk than kwarg-shaped ones, but prose-folded is safer still ("for X" rather than "(X)").

3. **Build-cache trap (process)**: the eval runner spawns the package's `dist/main.js` bin (per `package.json:bin`) via `resolvePackageBin("@neuve/local-agent")`, NOT source. Source-only changes don't reach the runtime until the package is rebuilt. The `feedback_no_test_only_injection_seams.md` discipline catches mock/stub drift but doesn't catch source-vs-built drift. Future-evals checklist: `pnpm --filter @neuve/local-agent build` before any sweep that exercises new source.
