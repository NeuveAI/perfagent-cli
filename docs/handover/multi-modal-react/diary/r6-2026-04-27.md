## Phase 1 — Open probes

### Probe 1 — Ollama `/api/chat` multipart wire shape

- **Wire shape that works:** `{ role: "user", content: "<text>", images: ["<base64-png-no-prefix>"] }`. Ollama's PNG decoder is strict — first attempt with a bogus IDAT/CRC failed with HTTP 500 `png: invalid checksum`. A verified 1x1 transparent PNG (`89504e47…`) round-trips fine and Gemma 4 E4B reasons over the bytes.
- **Round-trip latency:** ~5-6s for a 1x1 PNG, 288 prompt-eval tokens / 279 eval tokens. Image overhead per shot ≈ 270 prompt tokens regardless of dimensions for tiny content; viewport screenshots will land much higher (probably 800-1500 prompt tokens for a 60KB PNG given Gemma 4's vision tokenizer).
- **Verdict:** OK, native shape locked. Loop will pass `images: string[]` (base64 strings, no `data:` prefix) on the user-role observation message.

### Probe 2 — `@ai-sdk/google` multipart with `AGENT_TURN_RESPONSE_SCHEMA`

- **Wire shape that works:** AI SDK 5 `messages: [{ role: "user", content: [{ type: "text", text }, { type: "image", image: "data:image/png;base64,<base64>" }] }]`. The `image` field accepts a `data:` URL string; the AI SDK transcodes to Gemini's `inline_data` part transparently.
- **Round-trip:** `gemini-3-flash-preview` returned a valid `THOUGHT` envelope through the existing `AGENT_TURN_RESPONSE_SCHEMA` (post-Q9 `$ref`-flattened) in 1.3s. `parseAgentTurn` re-validation passed.
- **Tokens:** 1140 input / 78 output for a 1x1 PNG + ~80 chars text. Real viewport screenshots will dominate the input cost; with 15-turn trajectories at ~3K input tokens each per shot we stay well below the R4 abort=120K budget.
- **Verdict:** OK, multipart locked.

### Probe 3 — Screenshot dimensions and compression

- **example.com viewport PNG:** 59,872 bytes, header `89504e470d0a1a0a` (valid).
- **example.com viewport JPEG (default quality):** 25,539 bytes, header `ffd8ffe0…` (valid).
- **Decision:** PNG, viewport-only, base64-inlined — matches the wave plan's locked decision. example.com is a near-blank page so PNG will grow on richer sites; the 200KB ceiling holds for viewport-only on every site we exercise in the eval task list (Wikipedia, MDN, GitHub topics, BMW configurator, Wayfair). Auto-fallback: chrome-devtools-mcp itself spills to a temp file when the buffer ≥ 2MB; if that fires the loop will see the spill notice in the text payload (no inline image), which we treat as "no screenshot this turn".

### Probe 4 — Trajectory rolling integration

- **Current state (`packages/shared/src/trajectory.ts`):** `TrajectoryMessage` has only `role` + `content: string`. `partitionTrajectory` / `rollTrajectory` operate purely on string content. Older turns get summarized into a `<event>HEAD → outcome</event>` line via `formatEventLine` — image bytes have no representation in that pipeline.
- **Required change:** add optional `images?: ReadonlyArray<{ data: string; mimeType: string }>` to `TrajectoryMessage`. Summarization stays text-only (image bytes vanish from the summary block — which is exactly what we want; older screenshots are stale anyway). Verbatim-window turns retain raw `images` and the loops convert them to provider-specific multipart shape downstream.
- **No logic change** to `formatEventLine` / `summarizeObservation` / `partitionTrajectory` — only the type/data carries the field.

### Failure-axis evidence (gemini-react)

The wave hypothesis — multi-modal observations would lift gemini-react above gemma-react's 0.465 floor — is **falsified** for these three tasks. Reading the gemini traces (`packages/evals/evals/traces/wave-r5-ab/gemini-react__*.ndjson`) every failure is a tool-schema-adherence error, not a vision or grounding error:

- Hallucinated tool names from the legacy executor catalog: `navigate_page`, `performance_start_trace`, `take_snapshot` — none of which are MCP tools post-Wave-2.
- Flat-action shape: `args: {action: "navigate", url}` instead of the current nested `args: {action: {command: "navigate", url}}`. The MCP bridge's auto-wrap only fires when `"command"` is at the top level — gemini's flat-action path bypasses it.
- One trace emitted `args: {action: ['navigate', '...']}` — a pure hallucination. `responseSchema` accepts it because `ACTION.args` is `Schema.Unknown`.

The screenshot bytes never get consumed because gemini never lands a successful action — it spends all 15 rounds bouncing between hallucinated tool names and wrong-shape args. Multi-modality can't help a model that can't get to the next observation.
