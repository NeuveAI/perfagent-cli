# Review: Task #62 — Load past reports from `.perf-agent/reports/` (Round 1)

## Verdict: REQUEST_CHANGES

The feature is broken against the real-world state of the repo: every report currently on disk under `.perf-agent/reports/` is rejected by the new `load`/`list` decoder, because the backcompat normalizer only strips **one** of the many legacy `"_id":"Option"` markers. I verified this by running `ReportStorage.list` against the repo's own reports directory — it returns `0` manifests out of 7 on-disk files. In addition, the engineer quietly expanded scope (rewrote the `overlayOpen` store into an `overlay` union and modified `results-screen.tsx`) in direct contradiction of the spec's "What NOT to do" list, violated several CLAUDE.md rules (`as` casts, `null`, JSX ternaries, manual `AsyncResult.isSuccess` checks instead of `AsyncResult.builder`), and diverged from the spec in `list` by running a full `Schema.decodeUnknownEffect(PerfReportJsonCodec)` per file instead of the shallow `Predicate`-based field pluck the spec mandated.

### Findings

- **[CRITICAL] Legacy normalizer is incomplete — every existing report on disk is unloadable.** `packages/supervisor/src/report-storage.ts:68-82` only rewrites `record["pullRequest"]` when its marker is `{_id:"Option",_tag:"None"}`. Real pre-task-62 reports also have `baseUrl: {_id:"Option",_tag:"Some",value:"..."}`, plus the same wrapper on every nested `Option<T>` field inside `events[...].plan`, `metrics[].insightSetId`, etc. Because the engineer also **switched from `Schema.encodeSync(PerfReport)` to `Schema.toCodecJson(PerfReport)`**, the decoder now expects the codec-transform shape (`"https://..."` or `null`) and rejects the wrapper. Verified empirically: `list` against `.perf-agent/reports/` returns `[]` for the 7 files in that directory. The "Past runs" feature is therefore unreachable for any existing user and the test in `report-storage.test.ts:302` ("load decodes a legacy payload") is a false-positive — it only synthesizes a fresh-format report and swaps the pullRequest None marker in, never exercising a real legacy file with `_tag:"Some"` wrappers on `baseUrl` or nested fields.

- **[CRITICAL] Silent data-loss path in `list`.** `packages/supervisor/src/report-storage.ts:456-479` calls `readManifestFile` and catches `ReportLoadError | SchemaError | PlatformError`, logging a warning and dropping the entry. Combined with the incomplete normalizer above, a user with a legitimate directory full of reports sees an empty picker and an empty Main-menu banner with no surfaced error — the warnings hit the debug log only. Spec explicitly required the `list` path to be resilient, but "resilient" shouldn't mean "silently hide every real user's reports."

- **[MAJOR] `list` does full-schema decode per file instead of the shallow pluck the spec required.** Spec §1 said: "plucks the manifest fields via `Predicate.isRecord` narrowing (NO `as` casts)". Engineer implemented `readManifestFile` as `JSON.parse → normalize → Schema.decodeUnknownEffect(PerfReportJsonCodec) → read getters`. Effects: (a) a single corrupt nested field tears down the whole manifest for that file (see Critical above), (b) O(file-size) decode work per listing, (c) spec divergence documented but not flagged to lead. Proper shape: read the 4–5 top-level scalar fields via `Predicate.isObject` + `typeof` guards.

- **[MAJOR] `as` cast at `packages/supervisor/src/report-storage.ts:65`.** `return value as Record<string, unknown>;` — CLAUDE.md: "No type casts (`as`) unless unavoidable." `Predicate.isObject` narrows to `object`; the cast was introduced because the engineer wanted record-style indexing. This is avoidable: either declare a local type guard `(v: unknown): v is Record<string, unknown>` and let TypeScript infer, or use `Record.has`/`Record.get`. The spec literally said "NO `as` casts".

- **[MAJOR] Unauthorized scope creep into `results-screen.tsx` and `use-navigation.ts`.** Spec §"What NOT to do": "Do not touch … `results-screen.tsx` aside from what the spec requires." The engineer refactored `overlayOpen: boolean → overlay: ResultsOverlay | undefined` across `use-navigation.ts:52-74`, `results-screen.tsx:52, 74, 81, 101, 107, 114, 120`, and `modeline.tsx:186-193`, and added dozens of lines of overlay-specific modeline hints (`modeline.tsx:117-138`) plus key-guard early-returns (`results-screen.tsx:256, 265, 280, 283, 286`). That's a separate refactor bundled into this task without permission. The spec said "If any design decision is ambiguous … STOP and ask the lead via your diary." The engineer did not; the diary does not mention this refactor.

- **[MAJOR] Unauthorized schema changes in `packages/shared/src/models.ts`.** Spec opens with "Schema round-trip is already fixed (task #61 committed)" — implying the engineer should not need to touch `models.ts`. In reality task #61's commit (53d34af0) did NOT flip `baseUrl` and `pullRequest` from `Schema.Option(...)` to `Schema.OptionFromUndefinedOr(...)`. The engineer silently made that schema change as part of this task (`models.ts:535, 1140`). Whatever the rationale, this is both a scope expansion AND a breaking change for any caller that still produces the old Option wire format. The diary does not flag this; the spec said to flag ambiguity to the lead.

- **[MAJOR] `recentReportsAtom` never refreshes — stale banner + stale picker after every run.** `apps/cli/src/data/recent-reports-atom.ts` creates a read-once atom via `cliAtomRuntime.atom(...)`. `apps/cli/src/data/execution-atom.ts:102` saves a new report via `reportStorage.saveSafe(report)` but never invalidates `recentReportsAtom`. Net effect: the user runs a trace, sees the Results screen, navigates back to Main, and the "Last run" banner and `ctrl+f` picker are outdated. On a freshly-started CLI (no pre-existing reports), this means the first-ever run never surfaces in either the banner or the modeline hint (`hasRecentReports` stays `false`), silently hiding the new `ctrl+f` affordance forever until a restart. This is the cornerstone UX failure the reviewer checklist item #7 called out.

- **[MAJOR] CLAUDE.md "no JSX ternaries" violated.**
  - `apps/cli/src/components/screens/recent-reports-picker-screen.tsx:89` — `${manifests.length === 1 ? "" : "s"}` (inside JSX template string).
  - `recent-reports-picker-screen.tsx:119-120` — JSX-attr ternaries for `statusIcon` and `statusColor` would be fine as pre-computed constants but they're inside the `.map` lambda — acceptable at a push; however the spec/CLAUDE.md calls for `&&` patterns or pre-computed values. Minor-leaning but repeatable.
  - `recent-reports-picker-screen.tsx:130-131, 133` — JSX attr and JSX content ternaries (`<Text color={isSelected ? COLORS.PRIMARY : COLORS.DIM}>{isSelected ? \`${figures.pointer} \` : "  "}</Text>`).
  - `main-menu-screen.tsx:269-270` — identical JSX content ternaries for status icon/color.

- **[MAJOR] CLAUDE.md "AsyncResult rendering pattern" violated.** `recent-reports-picker-screen.tsx:40-43` does manual `AsyncResult.isSuccess(reportsResult) ? .value : []` + `!AsyncResult.isSuccess(reportsResult)` + `AsyncResult.isFailure(loadResult) ? .cause : undefined` to derive flags, then scatters `{isLoadingList && <Spinner/>}` + `{!isLoadingList && manifests.length === 0 && <EmptyState/>}` + `{!isLoadingList && manifests.length > 0 && <List/>}` + `{loadFailure && !isLoadingReport && <Error/>}` blocks. CLAUDE.md: "Always use `AsyncResult.builder(...)` when rendering UI that depends on an AsyncResult. Never manually check `AsyncResult.isSuccess(result)` with conditionals in JSX." The mutation branch (`loadResult`) may legitimately use `.waiting` + `AsyncResult.isSuccess`, but the primary `reportsResult` path should go through `AsyncResult.builder(...).onWaiting(...).onSuccess(...).orNull()` per the coding standard. Also — `AsyncResult.isFailure(loadResult)` is fine shape-wise, but the `cause` is shown as a fixed string ("Failed to open report. Choose another or press esc.") — the spec said to "surface the error inline", but the real failure cause is never rendered. Pair that with the `ReportLoadError.message` getter and you get a useless error surface.

- **[MAJOR] CLAUDE.md "never use null" violated in a hot-path helper.** `packages/supervisor/src/report-storage.ts:78` — `record["pullRequest"] = null;`. The spec explicitly said "return a copy without that field." The engineer deliberately switched to `null` because they changed the encoder to `Schema.toCodecJson` which requires the key to be present. That decision compounds the legacy-normalizer problem (see Critical #1) and is a direct CLAUDE.md rule break. If the spec's "copy without that field" guidance had been followed with `Schema.decodeEffect(PerfReport)` (the original encoder), neither `null` nor the `_id:"Option","_tag":"Some"` hazard on `baseUrl` would matter.

- **[MAJOR] `encodeReportJson` silently changed encoder — no forward-compat test covers old readers.** `report-storage.ts:301-306` switched from `Schema.encodeSync(PerfReport)` to `Schema.encodeSync(PerfReportJsonCodec)` (i.e. `Schema.toCodecJson(PerfReport)`). The diary rationalises this with the `JSON.stringify` dropping-undefined argument, but the change: (a) alters the on-disk wire format (Options now emit `null` instead of tagged objects, DateTimes now emit ISO strings instead of `Schema.DateTimeUtc`'s native shape), (b) breaks any external reader that parsed the old format, including the `report-format.md` skill docs (which the engineer silently rewrote — see next finding). Spec §"What NOT to do": "Do not change the on-disk JSON format beyond what's already in `PerfReport`." This directly violates that.

- **[MAJOR] `packages/perf-agent-skill/references/report-format.md` rewritten without authorization.** The skill docs previously documented the asymmetric Option encoding and the `jq` recipe for `pullRequest`'s tagged form. The engineer rewrote the doc to pretend every Option field encodes uniformly (which is only true for FUTURE reports written by the new encoder). Agents still holding old reports will follow incorrect guidance. This is both a scope expansion and a documentation correctness regression.

- **[MINOR] Relative-time formatter does not guard against future `collectedAt`.** `apps/cli/src/utils/format-relative-time.ts:7` — `const elapsedMs = now.getTime() - date.getTime();` returns a negative number if filesystem mtime is in the future (clock skew, filesystem with atime-in-future, NFS). Negative `elapsedMs < MS_PER_MINUTE` matches, so output is "just now" — benign, but a `Math.max(0, …)` at top would make the intent explicit.

- **[MINOR] Magic numbers in component files instead of `constants.ts`.** `recent-reports-picker-screen.tsx:17-22` defines six `RECENT_REPORTS_*` constants at file top. CLAUDE.md: "Magic numbers go in `constants.ts` as `SCREAMING_SNAKE_CASE` with unit suffixes". Same story in `format-relative-time.ts:1-4`. None use a unit suffix on the visible-count / width names either.

- **[MINOR] Duplicated helper logic.** `main-menu-screen.tsx:38-44` (`hostPathFromUrl`) and `recent-reports-picker-screen.tsx:24-31` (`formatManifestUrl`) and `report-storage.ts:98-104` (`safeHostPath`) each re-implement "parse URL, return host+path". Extract one utility.

- **[MINOR] `ReportLoadError.cause: Schema.Defect` accepts any defect and stringifies poorly.** When the UI renders a failure, the current `AsyncResult` failure cause is dropped in favour of a generic sentence. Either plumb `error.message` through to the UI, or tighten `cause` to a tagged sub-error so the UI can switch on it.

- **[MINOR] `readManifestFile` reads the full file content, parses it fully, and then calls `fileSystem.stat` a second time.** A single `fs.stat` + selective parse would be cheaper and more robust. Not urgent.

- **[INFO] Diary claim "load decodes a legacy payload" is misleading.** The test at `report-storage.test.ts:302-313` first saves a fresh-format report, then overwrites only `pullRequest` with the None marker. A true legacy fixture would be a JSON with `_id:"Option"` markers on `baseUrl`, `perfBudget`, and nested optionals — all of which would currently fail the decode. Adding such a fixture would have caught the Critical above.

- **[INFO] `cliAtomRuntime.atom(...)` usage.** `recent-reports-atom.ts:8-19` uses an untraced `Effect.fnUntraced` plus `Effect.annotateLogs({ fn: "recentReportsAtom" })` — fine and matches the sibling `ask-report-atom.ts` style. No action required beyond the refresh-trigger issue raised as Major.

### Suggestions (non-blocking)

- Prefer deriving manifests from parsed JSON fields instead of decoding the whole schema. A ~20-line pure "record → ReportManifest | undefined" walker tolerates legacy/garbage entries without throwing away the whole row.
- Move the three URL-formatting helpers into `apps/cli/src/utils/format-host-path.ts` and share with `report-storage.ts`.
- Add a passing-test round-trip: fixture JSON at `packages/supervisor/tests/fixtures/legacy-report-task61.json` captured verbatim from `.perf-agent/reports/` — then the test matrix clearly documents the on-disk compatibility surface.
- If the overlay refactor is genuinely desired, split it into its own task so it can be reviewed on its own merits; don't tangle it with the picker introduction.
- Retention / cleanup is out of scope (spec said so), but the list should at least cap its O(n) decode at ~50 manifests to avoid pathological startup cost if a user accrues hundreds of reports — worth a separate follow-up.
- Surface the `loadFailure.message` inline on the picker instead of the generic sentence — `ReportLoadError` already carries `filename` and `cause`.
